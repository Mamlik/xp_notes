# Конспекты МИФИ · Экстремальное программирование

Сайт-библиотека конспектов по компьютерным дисциплинам программы
«Экстремальное программирование» в МИФИ.

Текущий MVP:

- главная страница с карточками конспектов;
- публичная статическая витрина на GitHub Pages;
- каталог дисциплин и материалов в JSON;
- админка для загрузки HTML-конспектов;
- список открытых Pull Request в админке;
- добавление материалов через GitHub Pull Request;
- редактирование и удаление карточек через Pull Request;
- публикация только после ревью и merge.

## Архитектура

Проект устроен как небольшой Node.js-сервер без базы данных.

```text
index.html                 Главная страница библиотеки
admin.html                 Админка добавления конспектов через PR
server.js                  HTTP-сервер, статика, GitHub PR API
package.json               Скрипты запуска
.env.example               Пример переменных окружения
.github/workflows/
  deploy.yml               Автодеплой Node-сервера на VPS
  pages.yml                Публикация статической витрины на GitHub Pages
assets/
  app.js                   Рендер карточек и поиск на главной
  admin.js                 Логика формы админки
  catalog.js               JS-фолбэк каталога для открытия сайта как файла
  styles.css               Общие стили сайта
  mephi-logo-cropped.jpg   Логотип в шапке
data/
  lectures.json            Основной каталог дисциплин и конспектов
lectures/
  acos/                    Конспекты по АКОС
  algorithms/              Конспекты по алгоритмам
  distributed/             Конспекты по распределенным системам
```

Главная страница сначала пытается загрузить `data/lectures.json`.
Если сайт открыт как локальный файл и браузер блокирует `fetch`, используется
фолбэк из `assets/catalog.js`.

На боевом сервере источником правды считается `data/lectures.json`.

## GitHub Pages

Публичная часть сайта публикуется на GitHub Pages:

```text
https://mamlik.github.io/xp_notes/
```

Pages получает только статический артефакт:

- `index.html`;
- `assets/app.js`;
- `assets/catalog.js`;
- `assets/styles.css`;
- `assets/mephi-logo-cropped.jpg`;
- `data/`;
- `lectures/`.

Админка, серверный код и секреты не публикуются на GitHub Pages. Кнопка `Войти`
на публичной странице ведет на VPS-админку:

```text
http://213.171.29.190/admin.html
```

После merge в `main` запускаются два независимых деплоя:

- VPS-деплой обновляет Node-сервер и админку;
- Pages-деплой обновляет публичную статическую витрину.

## Как работает добавление конспекта

Админка доступна по адресу:

```text
/admin.html
```

Поток добавления:

1. Администратор вводит пароль из `ADMIN_PASSWORD`.
2. Выбирает дисциплину:
   - `acos`;
   - `algorithms`;
   - `distributed`.
3. Заполняет название, описание, теги и статус.
4. Загружает HTML-файл конспекта.
5. Браузер отправляет данные на `POST /api/admin/submit-lecture`.
6. Сервер проверяет пароль и валидирует payload.
7. Сервер через GitHub API создает новую ветку.
8. В ветку добавляется HTML-файл в `lectures/<course>/`.
9. Обновляется `data/lectures.json`.
10. Обновляется `assets/catalog.js`.
11. Создается Pull Request в GitHub.
12. Материал появляется на сайте только после merge PR.

Такой подход оставляет историю изменений в Git и не дает админке напрямую
ломать production.

## Управление существующими карточками

После входа в админке доступны три режима:

- `Добавить` — загружает новый HTML-конспект и создает PR на добавление.
- `Редактировать` — выбирает существующую карточку, меняет название, описание,
  теги, статус и при необходимости заменяет HTML-файл. Результат тоже уходит в
  Pull Request.
- `Удалить` — выбирает существующую карточку и создает PR, который удаляет
  запись из каталога и HTML-файл из `lectures/`.

В правой колонке админки показывается список открытых Pull Request. Это нужно,
чтобы создатель заявки мог вернуться после перезагрузки страницы и найти свой PR.

## Текущая валидация

Валидация находится в `server.js`, функция `validateLecturePayload`.

Сейчас проверяется:

- дисциплина должна быть одной из разрешенных: `acos`, `algorithms`,
  `distributed`;
- название должно быть не короче 3 символов;
- описание должно быть не короче 10 символов;
- HTML должен содержать полный документ с тегом `<html`;
- поддерживаются только файлы с расширением `.html`;
- имя файла очищается от опасных символов и путей;
- статус приводится к `published` или `draft`;
- пароль администратора проверяется сервером через заголовок
  `X-Admin-Password`.
- статический сервер не отдает скрытые файлы и папки, например `.env`, `.git`,
  `.github`.

Важно: сейчас HTML-конспект не санитизируется. Это осознанно для MVP, потому что
конспекты являются полноценными HTML-страницами со своими стилями. Безопасность
обеспечивается ревью Pull Request перед merge. Не стоит выдавать пароль админки
людям, которым нельзя доверять HTML-код.

Что можно усилить позже:

- проверять максимальный размер HTML-файла;
- запрещать `<script>` в пользовательских конспектах или помещать конспекты в
  sandboxed iframe;
- добавить GitHub OAuth вместо общего пароля;
- проверять уникальность `href` до создания PR;
- запускать CI-проверку JSON, ссылок и HTML в Pull Request.

## Локальный запуск

Нужен Node.js 20+.

```bash
npm start
```

По умолчанию сайт будет доступен на:

```text
http://127.0.0.1:8080/
```

## Переменные окружения

Создать `.env` из примера:

```bash
cp .env.example .env
```

Пример:

```env
PORT=8080
HOST=127.0.0.1
ADMIN_PASSWORD=your-admin-password
GITHUB_OWNER=Mamlik
GITHUB_REPO=xp_notes
GITHUB_TOKEN=github_pat_or_fine_grained_token
GITHUB_BASE_BRANCH=main
SERVER_PUBLIC_URL=http://213.171.29.190
```

`GITHUB_TOKEN` лучше делать fine-grained и выдавать доступ только к одному
репозиторию.

`SERVER_PUBLIC_URL` используется в описании Pull Request, чтобы давать прямую
ссылку на предпросмотр HTML-страницы из ветки PR до merge.

Нужные права токена:

- `Metadata: Read`;
- `Contents: Read and write`;
- `Pull requests: Read and write`.

Нельзя коммитить `.env` в репозиторий.

## VPS

Текущий деплой настроен так:

```text
project path: /home/mamlik/xp_notes
systemd service: xp-notes
node listen: 127.0.0.1:8080
nginx public port: 80
```

Полезные команды:

```bash
sudo systemctl status xp-notes --no-pager
sudo systemctl restart xp-notes
sudo journalctl -u xp-notes -f
sudo nginx -t
sudo systemctl reload nginx
```

После изменения `.env` нужен рестарт:

```bash
sudo systemctl restart xp-notes
```

## GitHub PR-flow

Сервер использует GitHub REST API:

1. читает `refs/heads/<GITHUB_BASE_BRANCH>`;
2. создает ветку `add-lecture/<timestamp>-<slug>`;
3. добавляет HTML-файл через Contents API;
4. обновляет `data/lectures.json`;
5. обновляет `assets/catalog.js`;
6. создает Pull Request.

Проверенный результат MVP: тестовый PR был создан успешно.

## Ручное добавление без админки

Если нужно добавить материал вручную:

1. Положить HTML-файл в нужную папку внутри `lectures/`.
2. Добавить запись в `data/lectures.json`.
3. Обновить `assets/catalog.js`.
4. Создать Pull Request.
5. Проверить сайт после merge.

Главную страницу `index.html` при этом редактировать не нужно.
