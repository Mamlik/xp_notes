📂Как пользователь взаимодействует с железом

С точки зрения userspace-программы никакого "железа" не видно напрямую. Всё взаимодействие с аппаратурой происходит через файлы — это основной принцип Unix.

Цепочка вызовов
1
Пользовательский код читает / пишет в "особый" файл (например /dev/tty, /dev/sda)
2
Ядро ОС Файловая система распознаёт, что файл — это устройство (device). Вызываются специальные колбеки драйвера.
3
Драйвер устройства Читает/пишет в "особую" память (MMIO) или использует порты ввода-вывода — и этим управляет реальным железом.
Ключевой принцип: пользовательский код никогда не дёргает железо напрямую — только через абстракцию файла и ядро.

🔐Кольца защиты (Protection Rings)

x86-архитектура определяет 4 уровня привилегий — кольца 0–3. Но на практике используются только два крайних:

Ring 0 Режим ядра (Kernel mode)

Полный доступ к железу, привилегированные инструкции, прямая работа с памятью.

Ring 1 / 2 Не используются

Задумывались для драйверов и сервисов ОС. Не прижились из-за сложности реализации и накладных расходов по скорости.

Ring 3 Режим пользователя (User mode)

Обычные программы. Нет прямого доступа к железу — только через системные вызовы.

Запомни: кольца 1 и 2 существуют на уровне спецификации Intel, но реальные ОС (Linux, Windows) их игнорируют. Вопрос «зачем они тогда есть?» часто проверяет понимание различия между архитектурной возможностью и практикой реальных ОС.
// 03

🔧Архитектура чипсета: Intel x58 IO Hub

Понимание физической компоновки важно для понимания задержек и производительности.

Историческая схема (Intel Core i7, ~2008)
  • Контроллер памяти (IMC) — интегрирован прямо в процессор. Цель: ускорить доступ к RAM, убрав лишнюю шину.
  • I/O Hub (IOH) — отдельная микросхема. Управляла PCI Express (GPU), USB, SATA и пр.
Современная схема (13-е поколение Intel Raptor Lake и т. д.)
  • Контроллер памяти — в процессоре (было и раньше).
  • PCI Express для GPU и NVMe — тоже интегрированы в процессор. IOH как отдельной микросхемы нет.
  • PCH (Platform Controller Hub) — остался для периферии: USB, SATA, медленный PCIe для Wi-Fi/звука/etc.
Почему это важно? Путь до памяти и до быстрых устройств (GPU, NVMe) стал короче → меньше задержки. Периферия по-прежнему идёт через PCH → она медленнее.

⚡3 механизма взаимодействия с аппаратурой

1. Порты ввода-вывода (I/O Ports)

Устаревший Отдельное адресное пространство, не пересекается с памятью

Специальные инструкции IN / OUT для чтения и записи. Применяются сейчас редко — только для очень старой периферии (COM-порты, PIC) и конфигурации PCI через 0xCF8/0xCFC.

2. Ввод/вывод через память (MMIO — Memory-Mapped I/O)

Основной современный способ Регистры устройства отображаются на физические адреса памяти

Запись по адресу = команда устройству. Чтение по адресу = состояние устройства.

Важно про кеширование при MMIO: процессор кеширует обращения к памяти — записи сначала идут в кеш, а не сразу на шину. Для устройств это катастрофа: команда "отправилась" в регистр, но до устройства не дошла. Решение — отключить кеширование для MMIO-страниц через PageTable (атрибуты страницы).
Режим кешаНазваниеПоведениеПрименение
WB Write Back Запись в кеш; в память — отложенно Обычная RAM — максимальная скорость
WT Write Through Запись и в кеш, и в память одновременно Редко; для согласованности без полного UC
UC Uncacheable Каждое обращение идёт прямо на шину MMIO-регистры устройств
WC Write Combining Запись не кешируется, но накапливается в буфере и сбрасывается пакетом Видеопамять (framebuffer) — нет случайного чтения
Трюк с буфером (framebuffer): держать копию экрана в RAM (быстро), а при обновлении — копировать во MMIO framebuffer. WC-режим позволяет пакетно сбрасывать изменения без штрафа за каждую запись.
volatile в драйверах

При работе с MMIO-регионами нужен volatile-доступ. Без него компилятор может: убрать "лишние" чтения (мол, значение не менялось), переставить операции, удалить запись (мол, результат нигде не используется). volatile говорит компилятору: "у этих операций есть побочные эффекты — не трогай!"

3. Прерывания (Interrupts)

Асинхронный механизм Устройство само сигнализирует процессору о готовности

CPU не опрашивает устройство в цикле (polling). Устройство завершает работу → взводит прерывание → CPU прерывает текущий поток → выполняет ISR (interrupt service routine) → возвращается.

🔄DMA и типы устройств Linux

DMA — Direct Memory Access

Зачем нужен DMA

При обычной передаче данных CPU вынужден сам копировать байты из/в устройство — это занимает процессорное время. DMA позволяет устройству самостоятельно писать в RAM без участия CPU.

  • CPU конфигурирует DMA-контроллер: откуда, куда, сколько байт.
  • Устройство записывает данные напрямую в RAM.
  • По завершению устройство поднимает прерывание — CPU узнаёт, что данные готовы.
  • CPU во время передачи полностью свободен.

Два типа устройств в Linux

Character Devices Символьные устройства

Примеры: /dev/tty, /dev/input/eventX

Доступ без кешей ОС — данные идут напрямую. Поток байт без структуры блоков.

Block Devices Блочные устройства

Примеры: /dev/sda, /dev/nvme0n1

Идут через block layer: очереди, планировщики, page cache, объединение запросов.

Page cache у блочных устройств — это буфер в RAM, хранящий недавно прочитанные/записанные блоки. Если файл читается повторно — данные из кеша, без обращения к диску. Именно поэтому повторное чтение одного файла в разы быстрее первого.

🔌PCI / PCIe — шина подключения устройств

Идентификация устройств

VendorID + DeviceID

Каждое PCI-устройство имеет пару 16-битных ID, по которым ОС находит нужный драйвер. Хранятся в базах данных:

  • Linux: /usr/share/hwdata/pci.ids
  • Команда для просмотра устройств: lspci
  • Онлайн-базы: pci-ids.ucw.cz и подобные

MSI-прерывания vs. IRQ-линии

Старый способ: IRQ-линии Физические провода прерываний

Устройство поднимает напряжение на линии → контроллер прерываний → CPU. Ограничено числом линий, проблема shared IRQ.

Современный: MSI Message Signaled Interrupts

Устройство пишет специальное сообщение в память по заранее настроенному адресу → CPU его читает. Меньше задержек, нет разделяемых линий, поддержка множества векторов на одно устройство.

Пайплайн обращения к PCI-устройству

1
Чтение конфигурации Через порты 0xCF8 (Config Address) и 0xCFC (Config Data). Записываем в 0xCF8 адрес нужного регистра конфигурации устройства, читаем результат из 0xCFC.
2
Получение MMIO-адресов или портов В конфигурационном пространстве PCI есть BAR (Base Address Registers) — они сообщают, по каким адресам памяти или портам "живут" регистры устройства.
3
Работа с устройством Обращаемся по найденным MMIO-адресам (через маппинг в виртуальное пространство) или читаем/пишем в порты.
Интересный факт: даже "современный" MMIO читается через старые порты 0xCF8/0xCFC на этапе discovery. Порты ввода-вывода умерли не полностью — они нужны для начального обнаружения PCI-устройств.

🎯Вопросы на понимание

Q01
Почему кольца 1 и 2 не используются в реальных ОС, хотя архитектура их поддерживает?
→ Задумывались для драйверов и сервисов ОС, но оказались слишком сложными в реализации и медленными. Проще работать с двумя уровнями: ядро (0) и пользователь (3).
Q02
Чем MMIO принципиально отличается от портов ввода-вывода?
→ Порты — отдельное адресное пространство (инструкции IN/OUT). MMIO — регистры устройства находятся в обычном адресном пространстве памяти. MMIO проще для программиста, удобнее кешировать и работает через обычные инструкции MOV.
Q03
Почему для MMIO-регионов нужно отключать кеширование? Что будет, если не отключить?
→ Без UC запись пойдёт в кеш процессора, а не сразу на шину. Устройство не получит команду. Более того, порядок сброса из кеша непредсказуем — команды к устройству могут прийти в неправильном порядке.
Q04
Зачем нужен режим кеша WC (Write Combining)? Чем он лучше UC для видеопамяти?
→ UC — каждая запись немедленно идёт на шину. WC — записи накапливаются и сбрасываются пакетом, что сильно увеличивает пропускную способность при последовательной записи (framebuffer). Видеопамять читается редко, но пишется часто и последовательно — WC идеален.
Q05
Для чего нужен volatile при работе с MMIO в драйвере?
→ Компилятор может оптимизировать "лишние" обращения к памяти: удалить запись, убрать повторное чтение, переставить операции. volatile запрещает такие оптимизации, гарантируя, что каждое чтение/запись реально идёт на шину.
Q06
Как работает DMA и зачем он нужен? В чём выигрыш по сравнению с обычной передачей через CPU?
→ CPU настраивает DMA-контроллер (буфер, размер). Устройство само копирует данные в RAM. CPU свободен. По готовности — прерывание. Выигрыш: CPU не занят побайтовой пересылкой, может выполнять другую работу.
Q07
В чём разница между character device и block device с точки зрения работы с ними в ядре?
→ Character device: доступ без кешей ОС, напрямую. Block device: через block layer (page cache, планировщики очередей, объединение запросов). Block-устройства выигрывают от кеширования и сортировки запросов.
Q08
Почему контроллер памяти был перенесён с северного моста (northbridge/IOH) внутрь процессора?
→ Чтобы сократить задержку доступа к RAM. Пока контроллер был на IOH, каждое обращение к памяти шло через внешнюю шину. После интеграции — прямой путь внутри чипа.
Q09
Чем MSI-прерывания лучше традиционных IRQ-линий?
→ IRQ: физические линии, ограниченное число, проблема shared IRQ (несколько устройств на одной линии → нужно опрашивать все). MSI: сообщение в память, нет ограничений по числу, каждое устройство/очередь может иметь свой вектор, меньше задержки.
Q10
Объясни пайплайн: как драйвер узнаёт, по каким адресам работать с PCI-устройством?
→ Через порты 0xCF8/0xCFC читается конфигурационное пространство PCI. Там находятся BAR (Base Address Registers) с MMIO-адресами или номерами портов ввода-вывода конкретного устройства.
Q11
Что такое page cache и зачем он у блочных устройств, но не у символьных?
→ Page cache — буфер в RAM, хранящий прочитанные/записанные блоки данных с диска. Повторное чтение — из памяти без обращения к диску. У character devices данные потоковые (tty, input events) — их кешировать нет смысла, нужна немедленная доставка.
Q12
Можно ли из userspace напрямую записать что-то в регистр устройства по MMIO-адресу?
→ Напрямую — нельзя. Физический MMIO-адрес нужно сначала отмапить в виртуальное адресное пространство (через /dev/mem или ioctl в драйвере). Ядро контролирует, кто может получить маппинг на физические адреса устройств.
