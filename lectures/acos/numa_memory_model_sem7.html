🧭Карта конспекта: что здесь изучается

Этот конспект про то, что на современных многопроцессорных машинах память уже нельзя воспринимать как одну простую коробку, куда все ядра одинаково быстро читают и пишут.

NUMA Память физически распределена

У каждого сокета есть своя локальная память. Доступ к чужой памяти возможен, но медленнее.

Reordering Операции могут менять порядок

И компилятор, и CPU могут переставлять чтения/записи ради скорости, если считают это безопасным.

Atomics Нужна управляемая синхронизация

Атомики дают атомарность и, в зависимости от режима, запрещают опасные переупорядочивания.

Главная идея конспекта: в однопоточном коде часто кажется, что инструкции выполняются ровно в том порядке, в котором написаны. В многопоточном коде это иллюзия. Между исходным кодом и реальной памятью стоят компилятор, микроархитектура CPU, кеши, store buffers, invalidation queues и протоколы когерентности.
// 01

🧩NUMA: зачем вообще разделять память

NUMA расшифровывается как Non-Uniform Memory Access, то есть «неодинаковый доступ к памяти». Это архитектура, где разные процессоры/сокеты имеют разную стоимость доступа к разным участкам RAM.

Мотивация

Если процессоров и ядер мало, можно представить одну общую память, к которой все обращаются примерно одинаково. Но при большом числе процессоров такая схема становится дорогой: все борются за один общий контроллер памяти, растут задержки, усложняется синхронизация, шины и контроллеры становятся узким местом.

UMA, простая модель:
CPU0 ─┐
CPU1 ─┼── общий контроллер памяти ── DRAM
CPU2 ─┘

NUMA, масштабируемая модель:
Socket 0: cores + L3 + memory controller ── local DRAM 0
│
├── interconnect ── Socket 1: cores + L3 + memory controller ── local DRAM 1
│
└── interconnect ── Socket 2: cores + L3 + memory controller ── local DRAM 2

Что такое NUMA node / NUMA socket

NUMA node — это обычно группа ядер, их локальные кеши/часть общего кеша и ближайшая к ним память. В серверной машине это часто совпадает с физическим сокетом процессора, поэтому говорят NUMA socket.

Локальная память Память своего NUMA-узла

Доступ быстрее: ядро обращается к DRAM через свой локальный контроллер памяти.

Удалённая память Память другого NUMA-узла

Доступ медленнее: запрос идёт через межсокетное соединение к другому контроллеру памяти.

Важно для КР: NUMA не значит, что ядро не может прочитать чужую память. Может. Но время доступа неравномерно: локальная память быстрее, удалённая медленнее.

Почему общие кеши локальны для сокета

Внутри сокета несколько ядер обычно делят последний уровень кеша, например L3. Но L3 одного сокета не является «бесплатно общим» для другого сокета. Если ядро на сокете 0 хочет данные, которые находятся ближе к сокету 1, придётся идти через межсокетную связь.

Практический вывод

Если поток работает с данными, которые лежат в памяти его NUMA-узла, всё хорошо. Если поток постоянно обращается к памяти другого узла, программа может тормозить даже при большом числе ядер.

🔗ccNUMA: cache-coherent NUMA

ccNUMA означает cache-coherent NUMA. Это NUMA-система, в которой кеши разных ядер/сокетов остаются когерентными: если одно ядро изменило данные, другие ядра не должны бесконечно читать старую версию из своих кешей.

Зачем нужна когерентность

Представь переменную x. Ядро A прочитало x и положило её в свой кеш. Потом ядро B изменило x. Если ничего не делать, ядро A может продолжить читать старое значение из кеша. Когерентность нужна, чтобы такие копии согласовывались.

1
Ядро читает память Кеш-линия попадает в локальный кеш ядра.
2
Другое ядро хочет записать в эту же кеш-линию Протокол когерентности должен убедиться, что старые копии у других ядер больше не считаются валидными.
3
Инвалидация или передача владения Другие кеши помечают свою копию как недействительную или меняют состояние кеш-линии.
Ловушка: cache-coherent не означает «быстро». ccNUMA гарантирует согласованность кешей, но удалённые обращения и межсокетные инвалидации всё равно дорогие.
// 03

🧠Compiler memory reordering

Compiler reordering — это переупорядочивание операций компилятором. Компилятор не обязан сохранять буквальный порядок строк исходного кода, если в рамках однопоточной логики результат не меняется.

Почему компилятор вообще переставляет операции

Компилятор оптимизирует код: убирает лишние чтения, держит значения в регистрах, меняет порядок инструкций, чтобы лучше использовать конвейер CPU. Для обычного однопоточного кода это почти всегда полезно.

// Наивное ожидание программиста:
data = 42;
ready = 1;

// В другом потоке:
if (ready) {
    use(data);
}

Проблема: если data и ready — обычные переменные без синхронизации, компилятор и CPU могут обращаться с ними так, что другой поток увидит ready = 1, но ещё не увидит корректный data = 42.

Барьер памяти для компилятора

Конструкция asm volatile ("" ::: "memory") — это compiler barrier, то есть барьер для компилятора.

asm volatile ("" ::: "memory")

Пустая inline-assembly инструкция ничего не выполняет на CPU, но часть "memory" говорит компилятору: «считай, что память могла измениться; не переставляй обращения к памяти через этот барьер».

Критически важно: compiler barrier не является CPU barrier. Он запрещает переупорядочивание компилятором, но не обязан сбрасывать store buffer, ждать инвалидаций или запрещать аппаратные переупорядочивания.

Ключевое слово volatile

volatile говорит компилятору, что обращение к объекту имеет наблюдаемый побочный эффект и его нельзя просто выкинуть или заменить старым значением из регистра.

Что volatile делает Влияет на оптимизации компилятора

Заставляет реально выполнять чтения/записи volatile-объекта и не считать их обычными лишними обращениями.

Что volatile не делает Не заменяет атомики и мьютексы

Не гарантирует атомарность сложных операций и не задаёт полноценную межпоточную синхронизацию в C/C++.

Типичная ловушка: volatile int x не делает x++ атомарным. Операция x++ всё равно раскладывается на read → modify → write.
// 04

⚙️CPU memory reordering

Даже если компилятор не переставил операции, это может сделать сам процессор. Современные CPU выполняют инструкции не строго «по одной сверху вниз», а стараются выполнять их максимально эффективно.

Store buffers

Store buffer — буфер записей. Когда ядро выполняет запись в память, оно не всегда ждёт, пока запись физически дойдёт до кеша/памяти и станет видна всем ядрам. Запись может временно попасть в store buffer, а ядро продолжит выполнять следующие инструкции.

Зачем нужен store buffer

Запись в общую память может быть дорогой: надо получить владение кеш-линией, инвалидировать копии у других ядер, дождаться протокола когерентности. Store buffer позволяет ядру не простаивать всё это время.

Core 0:
store x = 1 ── попадает в store buffer
load y ── может выполниться до того, как x станет виден другим ядрам

Следствие: другие ядра могут временно не видеть запись x = 1.

Invalidation queues

Invalidation queue — очередь входящих сообщений об инвалидации кеш-линий. Когда другое ядро записывает в кеш-линию, твоя старая копия должна стать недействительной. Но обработка таких сообщений тоже может буферизоваться.

Интуиция: store buffer ускоряет собственные записи, а invalidation queue позволяет не останавливаться мгновенно на обработке всех чужих инвалидаций. Оба механизма полезны для скорости, но усложняют модель памяти.

4 типа переупорядочивания

ТипЧто означаетПримерПочему опасно
StoreStore Две записи могут стать видимыми в другом порядке x=1; y=1; Другой поток может увидеть y=1, но ещё не увидеть x=1.
StoreLoad Запись может быть отложена, а последующее чтение выполнено раньше x=1; r=y; Классическая проблема store buffer; часто самый дорогой порядок для запрета.
LoadStore Чтение и последующая запись могут наблюдаться не так, как ожидается r=x; y=1; Опасно в алгоритмах синхронизации и lock-free коде.
LoadLoad Два чтения могут быть фактически выполнены/увидены не в исходном порядке r1=x; r2=y; Можно увидеть новый указатель, но старые данные по нему, если нет acquire-барьера.

CPU memory barriers: mfence, lfence, sfence

Аппаратные барьеры памяти — это инструкции, которые ограничивают переупорядочивание на уровне CPU.

mfence Memory fence

Полный барьер для операций памяти. Грубо: не даёт чтениям/записям пересекать границу барьера.

lfence Load fence

Барьер для чтений. Используется, когда важно упорядочить load-операции.

sfence Store fence

Барьер для записей. Используется, когда важно упорядочить store-операции.

Важно: барьеры памяти не «делают код быстрее». Наоборот, они часто замедляют выполнение, потому что запрещают CPU некоторые оптимизации. Их ставят не ради скорости, а ради корректности.
// 05

🔩Сложные операции: read → modify → write

Многие операции, которые в исходном коде выглядят как одна команда, внутри процессора раскладываются на несколько шагов. Общая схема: read → modify → write, сокращённо RMW.

1
Read Прочитать старое значение из регистра/кеша/памяти.
2
Modify Изменить значение внутри ядра: прибавить, сравнить, заменить и т. д.
3
Write Записать новое значение обратно.

Почему x++ не атомарен

// Два потока одновременно делают:
x++;

// На уровне идеи это:
tmp = x;      // read
tmp = tmp+1;  // modify
x = tmp;      // write

Если два ядра одновременно прочитали старое значение x = 10, оба посчитали 11 и оба записали 11, то один инкремент потерялся. Поэтому x++ без синхронизации — не атомарная операция.

xadd и cmpxchg

xadd Exchange and add

Атомарноподобная RMW-инструкция: взять старое значение, прибавить новое, вернуть старое. Часто используется для атомарных счётчиков.

cmpxchg Compare and exchange

Сравнить текущее значение с ожидаемым; если совпало — заменить. База для многих lock-free алгоритмов.

Префикс lock

Префикс lock перед инструкцией нужен, чтобы гарантировать атомарность сложной RMW-операции относительно других ядер.

Что делает lock концептуально

Он заставляет процессор выполнить RMW-операцию так, чтобы другие ядра не могли «вклиниться» своими микрооперациями к этой же памяти между read, modify и write. На современных CPU это обычно связано не с буквальной блокировкой всей шины, а с эксклюзивным владением кеш-линией и протоколом когерентности.

Ловушка: одна ассемблерная инструкция не всегда означает одну неделимую операцию на уровне памяти. Если инструкция делает RMW, без lock другие ядра могут наблюдать конфликтующие действия.
// 06

⚛️Atomics: зачем они нужны в высокоуровневых языках

Атомарные операции в языках вроде C/C++ нужны не только для того, чтобы «операция не разорвалась пополам». Их роль шире.

1 Запрещают опасные оптимизации компилятора

Компилятор обязан учитывать правила выбранной memory model.

2 Запрещают нужные виды CPU reordering

Генерируются инструкции/барьеры, которые заставляют CPU соблюдать нужный порядок.

3 Гарантируют атомарность сложных операций

Например, атомарный инкремент или compare-exchange не теряет обновления при гонке потоков.

Формулировка для ответа: атомики — это договор между программистом, компилятором и процессором о том, какие операции должны быть атомарными и какие переупорядочивания запрещены.

Relaxed

Relaxed — самый слабый режим. Он гарантирует атомарность самой операции, но почти не гарантирует порядок относительно других операций памяти.

Когда relaxed может быть достаточно

Например, для счётчика статистики: важно, чтобы инкременты не терялись, но не важно, в каком порядке относительно других данных они видны.

Ловушка: relaxed atomic не означает «обычная переменная». Атомарность есть. Но использовать relaxed как флаг готовности данных обычно нельзя: порядок публикации данных он не гарантирует.

Acquire / Release

Acquire/Release — модель для передачи данных между потоками через флаг, указатель, очередь, lock-free структуру и т. д.

Release Публикация данных

Все записи до release-записи не должны «уехать» после неё. Поток как бы говорит: «данные готовы».

Acquire Получение данных

Все чтения после acquire-чтения не должны «уехать» до него. Поток как бы говорит: «я увидел флаг, теперь читаю данные».

// Поток-писатель:
data = 42;
ready.store(true, memory_order_release);

// Поток-читатель:
if (ready.load(memory_order_acquire)) {
    // здесь должен быть виден data = 42
    use(data);
}
Интуиция: release закрывает «пакет» записей, acquire открывает этот пакет на другой стороне. Если acquire увидел значение, записанное release-операцией, данные до release становятся видны после acquire.

Sequentially consistent

Sequentially consistent, или seq_cst, — самый строгий и обычно самый понятный режим. Он гарантирует атомарность и глобальный порядок атомарных операций, который согласуется с порядком внутри каждого потока.

Почему seq_cst удобен

О нём проще рассуждать: можно представить, что все seq_cst-атомики выстроены в одну общую очередь. Но за простоту иногда приходится платить производительностью.

Memory orderАтомарностьПорядокТипичный смысл
Relaxed Да Почти не гарантируется Счётчики, статистика, независимые флаги без публикации данных.
Acquire Да Запрещает последующим операциям уехать до acquire Чтение флага/указателя перед использованием опубликованных данных.
Release Да Запрещает предыдущим операциям уехать после release Публикация данных через флаг/указатель.
Seq_cst Да Строгий глобальный порядок атомиков Самый простой для рассуждений, но потенциально дороже.

🧪Как всё это связано: от NUMA до atomics

Темы семинара связаны одной идеей: чем больше ядер и чем сложнее память, тем сложнее сделать вид, что всё работает как в простом однопоточном коде.

1
NUMA Память распределена физически, локальный доступ быстрее удалённого.
2
Кеши и ccNUMA Кеши должны оставаться согласованными, даже если данные гуляют между сокетами.
3
Store buffers / invalidation queues CPU буферизует записи и инвалидации, чтобы не простаивать.
4
Memory barriers Когда порядок критичен, нужны барьеры, которые ограничивают оптимизации.
5
Atomics Высокоуровневый способ получить атомарность и нужную модель видимости памяти.
Главная ошибка новичка: думать, что если один поток написал data, а потом ready = true, то другой поток, увидев ready, автоматически увидит и data. Без корректной синхронизации это не гарантировано.
// 08

🛠️Практические советы

NUMA-aware Держать данные рядом с потоком

Горячие данные лучше размещать в памяти того NUMA-узла, где выполняется поток, который с ними работает.

Affinity Не гонять потоки между сокетами

Миграция потока может привести к тому, что его данные останутся на старом узле, а он начнёт читать их удалённо.

Atomics wisely Выбирать минимально достаточный memory order

Seq_cst проще, но может быть дороже. Relaxed быстрее, но не подходит для публикации данных.

No volatile sync Не использовать volatile как замену синхронизации

Для межпоточного кода нужны атомики, мьютексы или другие корректные примитивы.

Для низкоуровневого кода: важно понимать, где нужен compiler barrier, где CPU fence, где атомик, а где достаточно обычной переменной под защитой мьютекса.

📌Мини-шпаргалка

  • NUMA — память распределена по узлам; локальная память быстрее удалённой.
  • NUMA node/socket — группа ядер + локальный L3/часть кешей + локальный DRAM-контроллер.
  • ccNUMA — NUMA, где кеши разных ядер/сокетов остаются когерентными.
  • Compiler reordering — компилятор может переставлять обращения к памяти, если в однопоточной модели это не меняет результат.
  • asm volatile("" ::: "memory") — барьер для компилятора, не аппаратный fence.
  • volatile — не атомик и не синхронизация между потоками.
  • Store buffer — запись может быть отложена и стать видимой другим ядрам позже.
  • Invalidation queue — обработка сообщений об инвалидации кеш-линий может буферизоваться.
  • mfence/lfence/sfence — аппаратные барьеры для ограничения CPU reordering.
  • RMW — read → modify → write; без атомарности такие операции могут конфликтовать.
  • lock — префикс, делающий RMW-инструкцию атомарной относительно других ядер.
  • Relaxed — атомарность без сильного порядка.
  • Acquire/Release — классическая модель публикации и получения данных.
  • Seq_cst — самый строгий и самый простой для рассуждений порядок.
// вопросы на понимание

🎯Вопросы на понимание

Q01
Что означает NUMA и чем NUMA отличается от UMA?
→ UMA: все процессоры имеют примерно одинаковую стоимость доступа к памяти. NUMA: память распределена по узлам, локальная память быстрее удалённой.
Q02
Почему при большом количестве процессоров выгодно разделять память по сокетам?
→ Чтобы уменьшить конкуренцию за один общий контроллер памяти/шину и лучше масштабировать систему. Каждый сокет получает свой контроллер DRAM.
Q03
Может ли ядро одного NUMA node обращаться к памяти другого node?
→ Да, может. Но это удалённый доступ: он идёт через межсокетное соединение и обычно медленнее локального.
Q04
Что означает ccNUMA?
→ Cache-coherent NUMA: память неравномерная по стоимости доступа, но кеши разных ядер/сокетов поддерживаются согласованными.
Q05
Если NUMA cache-coherent, значит ли это, что удалённая память такая же быстрая, как локальная?
→ Нет. Когерентность — про корректность видимости кешированных данных, а не про одинаковую скорость доступа.
Q06
Что такое compiler memory reordering?
→ Компилятор может менять порядок чтений/записей, если считает, что в рамках однопоточного выполнения результат не изменится.
Q07
Что делает asm volatile ("" ::: "memory")?
→ Это compiler barrier. Он запрещает компилятору переупорядочивать обращения к памяти через эту точку, но сам CPU-инструкций барьера не выполняет.
Q08
Почему compiler barrier не равен CPU memory barrier?
→ Compiler barrier влияет только на генерацию кода компилятором. CPU всё ещё может использовать store buffer, out-of-order execution и другие аппаратные механизмы.
Q09
Делает ли volatile переменную атомарной?
→ Нет. volatile запрещает некоторые оптимизации компилятора, но не гарантирует атомарность RMW-операций и не задаёт межпоточную синхронизацию.
Q10
Почему volatile int x; x++; не является атомарным?
→ Потому что инкремент — это read → modify → write. Два потока могут прочитать одно старое значение и потерять один инкремент.
Q11
Что такое store buffer и зачем он нужен?
→ Буфер записей ядра. Позволяет CPU продолжать работу, не дожидаясь, пока запись станет видна всем ядрам через кеш-когерентность.
Q12
Как store buffer может сломать наивные ожидания программиста?
→ Поток может выполнить запись, затем чтение, но запись ещё будет сидеть в store buffer и не станет видна другим ядрам.
Q13
Что такое invalidation queue?
→ Очередь сообщений о том, что кеш-линия должна быть инвалидирована. Она помогает производительности, но добавляет задержки в видимость изменений.
Q14
Назови четыре типа memory reordering.
→ StoreStore, StoreLoad, LoadStore, LoadLoad.
Q15
Какой тип reordering часто считается особенно неприятным?
→ StoreLoad: запись может быть отложена, а последующее чтение выполнено раньше с точки зрения видимости.
Q16
Чем отличаются mfence, lfence, sfence?
→ mfence — полный memory fence; lfence — барьер для load; sfence — барьер для store.
Q17
Что означает RMW?
→ Read → Modify → Write: прочитать значение, изменить его, записать обратно. Пример: инкремент, xadd, cmpxchg.
Q18
Зачем нужен префикс lock?
→ Чтобы сделать RMW-инструкцию атомарной относительно других ядер: никто не должен вклиниться между read, modify и write к той же памяти.
Q19
Что делает cmpxchg и почему он важен?
→ Compare-and-exchange: если текущее значение равно ожидаемому, заменить его новым. Это базовая операция для lock-free алгоритмов.
Q20
Какие три функции атомарных операций в C/C++?
→ Запрещают нужные переупорядочивания компилятором, запрещают нужные переупорядочивания CPU, гарантируют атомарность сложных операций.
Q21
Что гарантирует relaxed atomic?
→ Атомарность самой операции, но не общий порядок относительно других операций памяти.
Q22
Почему relaxed atomic не подходит как флаг готовности данных?
→ Потому что другой поток может увидеть флаг, но не обязан увидеть все записи данных, которые логически были до установки флага.
Q23
Что такое release?
→ Операция публикации: записи до release не должны переехать после неё. Используется, чтобы показать другим потокам, что данные готовы.
Q24
Что такое acquire?
→ Операция получения: чтения/записи после acquire не должны переехать до него. Если acquire увидел release, можно читать опубликованные данные.
Q25
Чем seq_cst отличается от acquire/release?
→ Seq_cst задаёт более строгий глобальный порядок всех sequentially consistent атомарных операций. Acquire/release задаёт направленную синхронизацию публикации и получения.
Q26
Почему барьеры памяти могут ухудшить производительность?
→ Они ограничивают оптимизации CPU, могут заставлять ждать завершения буферизованных операций и мешают out-of-order выполнению.
Q27
Почему обычный мьютекс часто проще и безопаснее ручных atomics?
→ Мьютекс уже содержит нужную синхронизацию и понятную модель владения. Ручные atomics требуют точно понимать memory order и легко ломаются.
Q28
Главная мысль конспекта одним предложением?
→ В многопроцессорных системах порядок и видимость операций памяти не очевидны; для корректности нужны NUMA-aware размещение, барьеры и атомарные операции с правильной memory model.
