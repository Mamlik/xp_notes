🧩Зачем вообще нужен IPC

IPC — inter-process communication, то есть способы обмена данными между процессами. Процессы в ОС специально изолированы друг от друга: у каждого своё виртуальное адресное пространство, свои дескрипторы, свои права. Это повышает безопасность и устойчивость, но создаёт вопрос: как им обмениваться информацией?

Главная идея: IPC — это контролируемая “дырка” в изоляции процессов. ОС даёт специальные механизмы, через которые процессы могут передавать байты, сообщения, события, память или даже файловые дескрипторы.
Поток байт Pipe / socket stream

Данные идут подряд, как в файл: прочитал N байт — не обязательно совпадает с границей “сообщения”.

Сообщения Message queue / datagram

Передаются отдельные пакеты: одно сообщение отправили — одно сообщение получили.

Общая память Shared memory

Данные не копируются через ядро каждый раз. Процессы видят один и тот же участок памяти.

📁Файлы / файловая система как IPC

Самый очевидный способ обмена: один процесс пишет в файл, другой читает. Это работает, но для IPC обычно плохо подходит, потому что файловая система исторически предназначена для долговременного хранения данных, а не для быстрого обмена событиями между живыми процессами.

✅ Когда файл подходит
  • Нужно сохранить результат надолго.
  • Процессы не обязаны работать одновременно.
  • Обмен редкий и не критичен по задержкам.
  • Нужна простая отладка: файл можно открыть глазами.
❌ Почему плохо для IPC
  • Нужно самому решать синхронизацию доступа.
  • Неудобно ждать появления новых данных.
  • Есть накладные расходы ФС, page cache, metadata.
  • Нет естественных границ “сообщений”.
Ловушка: “в Unix всё файл” не означает, что обычный файл — хороший IPC. Удобная файловая абстракция есть у многих IPC-механизмов, но семантика у них другая.
// 03

🚰Pipes и named pipes

Pipe — это буфер байтов в ядре с FIFO-семантикой: что первым записали, то первым прочитали. Он похож на “трубу”: один процесс пишет в один конец, другой читает из другого.

Обычный pipe

Создаётся системным вызовом pipe(). Обычно используется между родственными процессами: родитель создал pipe, затем сделал fork(), и оба процесса унаследовали нужные файловые дескрипторы.

пример: cmd1 | cmd2 в shell
Named pipe / FIFO

Создаётся в файловой системе, например через mkfifo. В отличие от обычного pipe, имеет имя в ФС, поэтому к нему могут подключаться неродственные процессы. При этом данные всё равно не “хранятся как файл”: FIFO — это специальный объект ядра.

пример: mkfifo /tmp/myfifo
Pipe: поток байт с FIFO-семантикой
Важно: pipe — это именно поток байтов. Если writer записал “сообщение” из 100 байт, reader может прочитать 40 байт, потом 60 байт. Границы логических сообщений приложение обычно должно восстанавливать само.
// 04

⚡Signals

Сигнал — это асинхронное уведомление процессу или конкретному потоку: “произошло событие, отреагируй”. Это не полноценный канал передачи данных, а скорее interrupt-like механизм на уровне процесса.

Главная идея: сигнал не “летит” напрямую из одного процесса в другой. Отправитель делает системный вызов или событие возникает в ядре, после чего ядро помечает сигнал как pending у целевого процесса/потока и позже доставляет его в безопасной точке выполнения.

Кто может отправить сигнал

Другой процесс kill(pid, sig)

Несмотря на название, kill не обязательно “убивает”: он просто просит ядро послать сигнал процессу или группе процессов.

Сам процесс raise(sig), alarm()

Процесс может послать сигнал самому себе, например для таймера, отмены операции или тестирования обработчика.

Ядро Исключения и события

Деление на ноль, segfault, завершение дочернего процесса, ввод с терминала, таймеры — всё это может породить сигнал.

Что именно происходит при отправке сигнала

1
Появляется источник сигналаНапример, процесс A вызывает kill(B, SIGTERM), пользователь нажимает Ctrl+C, CPU ловит page fault, или у процесса истекает таймер.
2
Управление попадает в ядроДля kill это системный вызов. Для ошибки памяти/деления — аппаратное исключение. Для терминала/таймера — событие от подсистемы ядра.
3
Ядро проверяет праваОбычно нельзя слать произвольные сигналы чужим процессам без подходящих прав. Например, обычный пользователь не может просто послать SIGKILL системному демону root.
4
Сигнал становится pendingЯдро записывает в структуру процесса/потока, что сигнал ожидает доставки. Он ещё не обязательно выполнен прямо сейчас.
5
Учитывается signal maskЕсли сигнал заблокирован через sigprocmask/pthread_sigmask, он остаётся pending до разблокировки. SIGKILL и SIGSTOP заблокировать нельзя.
6
Доставка при возврате в userspaceКогда поток возвращается из syscall/interrupt/trap в пользовательский режим, ядро проверяет pending-сигналы и решает, что делать.
7
Default action или handlerЕсли обработчик не задан — выполняется действие по умолчанию: завершить, остановить, игнорировать и т.д. Если задан sigaction, ядро подготавливает вызов handler-а.
Доставка сигнала: не прямой вызов, а работа через ядро

Как запускается signal handler

Handler — это не отдельный поток

Если процесс установил обработчик через sigaction, ядро не создаёт новый поток. Вместо этого оно меняет пользовательский контекст так, чтобы при возврате из ядра управление попало в функцию-обработчик. Обычно на пользовательском стеке создаётся специальный signal frame: там сохраняется старый контекст регистров, маска сигналов и адрес возврата через sigreturn. Когда handler заканчивается, sigreturn возвращает поток туда, где он был прерван.

пример установки обработчикаC
static volatile sig_atomic_t stop = 0;

static void on_sigterm(int signo) {
    stop = 1; // безопасно: простая запись в sig_atomic_t
}

int main() {
    struct sigaction sa = {0};
    sa.sa_handler = on_sigterm;
    sigemptyset(&sa.sa_mask);
    sigaction(SIGTERM, &sa, NULL);

    while (!stop) {
        // main loop
    }
}
Ловушка: из signal handler нельзя безопасно вызывать всё подряд. Например, printf, malloc, взятие mutex-а — плохая идея, потому что сигнал мог прервать код внутри этих же функций. В обработчике обычно ставят флаг типа sig_atomic_t или пишут байт в pipe/eventfd.

Process-directed vs thread-directed signals

Процессу kill(pid, sig)

Сигнал адресован процессу в целом. В многопоточной программе ядро выбирает один поток, который не блокирует этот сигнал.

Конкретному потоку pthread_kill(thread, sig)

Сигнал адресован выбранному thread-у. Это важно, потому что маска сигналов в POSIX фактически хранится на уровне потока.

Обычные и real-time signals

ТипОчередьДанныеПример
Обычные сигналыЧасто склеиваются: если SIGTERM уже pending, второй такой же может не добавить новую запись.Обычно только номер сигнала.SIGINT, SIGTERM, SIGCHLD
Real-time signalsМогут ставиться в очередь в порядке доставки.Через sigqueue можно передать маленькое значение.SIGRTMIN + n

Принудительное завершение процесса. Нельзя обработать, проигнорировать или заблокировать.

SIGTERM Вежливое завершение

Процесс может поймать сигнал, закрыть файлы, сохранить состояние и выйти сам.

SIGHUP Часто — reload

Для демонов часто означает “перечитай конфигурацию”, хотя исходно связан с потерей терминала.

Что передаёт сигнал

Базово сигнал передаёт очень мало информации: номер события. Поэтому signals хороши для уведомлений “что-то случилось”, но плохи для передачи больших данных. Для структурированных данных выбирают pipe/message queue/socket/shared memory, а сигнал иногда используют только как “пинок”: проснись и прочитай данные оттуда.

Ловушка: сигнал — это не очередь сообщений общего назначения. Обычные сигналы могут склеиваться, handler выполняется асинхронно относительно основного кода, а полезная нагрузка минимальна.

📬Message queues

Message queue — IPC-механизм для передачи дискретных сообщений, а не непрерывного потока байт. В POSIX/Linux такие очереди существуют в отдельной виртуальной файловой системе /dev/mqueue.

Плюс Сохраняются границы сообщений

Если отправили одно сообщение, получатель читает именно сообщение, а не произвольный кусок байтового потока.

Плюс Не привязаны к процессу

Очередь — отдельный объект ядра. Процессы могут открывать её по имени.

Минус Есть лимиты

Размер сообщения, число сообщений и права доступа ограничены настройками ОС.

Минус Копирование через ядро

Для больших потоков данных shared memory может быть эффективнее.

Message queue: очередь отдельных сообщений

🧠Shared memory

Shared memory — механизм, при котором несколько процессов маппят один и тот же объект памяти в свои виртуальные адресные пространства. В Linux POSIX shared memory обычно живёт в виртуальной файловой системе /dev/shm.

1
Создать объект shared memoryНапример, через POSIX shm_open(). Получаем файловый дескриптор.
2
Задать размерОбычно через ftruncate(), чтобы объект имел нужный размер.
3
Сделать mmapКаждый процесс вызывает mmap() и получает адрес в своём виртуальном пространстве.
4
Синхронизировать доступ вручнуюНапример, через mutex/semaphore/futex/atomics. Shared memory сама по себе не защищает от гонок.
Shared memory: разные виртуальные адреса → один физический объект
Главная опасность: shared memory даёт общую память, но не даёт автоматической синхронизации. Если два процесса одновременно пишут в одну структуру без протокола доступа — получатся race conditions, torn reads, потерянные обновления и неконсистентное состояние.
Схема POSIX shared memoryC / pseudo
int fd = shm_open("/my_shm", O_CREAT | O_RDWR, 0600);
ftruncate(fd, SIZE);
void *p = mmap(NULL, SIZE, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

// В p теперь общий регион памяти.
// Но нужен mutex/semaphore/atomic-протокол для безопасного доступа.

🔌Сокеты

Socket — универсальный интерфейс обмена данными. Важно различать сетевые сокеты и Unix domain sockets: интерфейс похожий, но путь данных и накладные расходы разные.

TCP / UDP / сетевые сокеты

Когда использовать

Когда процессы могут находиться на разных машинах или нужна стандартная сетевая коммуникация. TCP даёт поток байт с надёжной доставкой и порядком. UDP даёт датаграммы без гарантий доставки.

  • TCP: соединение, поток байт, доставка по порядку, повторы при потере.
  • UDP: отдельные датаграммы, меньше гарантий, меньше накладных расходов.
Важно: если процессы общаются только на одной машине, сетевые TCP/UDP-сокеты часто избыточны: они проходят через сетевые абстракции, имеют больше накладных расходов и ограничений. Часто лучше Unix domain sockets или shared memory.

Unix domain sockets

Unix domain sockets дают похожий socket-интерфейс, но не используют настоящий сетевой интерфейс. Данные передаются через память ядра внутри одной машины.

Быстрее локально Без IP-сети

Нет маршрутизации через внешний сетевой стек как для межмашинной коммуникации.

SCM_RIGHTS Передача FD

Можно передать файловый дескриптор другому процессу — очень важная Unix-фича.

Credentials Права процесса

Можно получать информацию о uid, gid, pid peer-процесса.

Как выбирать IPC на практике

1
Процессы на разных машинах?Да → TCP/UDP. Нет → локальный IPC обычно эффективнее.
2
Нужно передавать большие объёмы данных?Да → shared memory + синхронизация. Нет → socket/message queue/pipe.
3
Нужны границы сообщений?Да → message queue, datagram socket, свой framing поверх stream. Нет → pipe/TCP/stream UDS.
4
Нужно передавать файловые дескрипторы?Да → Unix domain sockets.
5
Нужно просто уведомить процесс?Да → signal может быть достаточно, но не стоит пихать туда данные.
Мини-шпаргалка: pipe — поток байт; message queue — сообщения; shared memory — быстро и опасно без синхронизации; signal — уведомление, не данные; TCP/UDP — сеть; Unix domain socket — локальный socket API + специальные Unix-фичи.

🎯Вопросы на понимание

Q01
Почему обычные файлы плохо подходят для IPC?
→ Они предназначены для хранения, а не обмена: нет удобного ожидания сообщений, нет границ сообщений, нужна синхронизация, есть накладные расходы ФС.
Q02
Чем pipe отличается от обычного файла?
→ Pipe — специальный буфер в ядре с FIFO-семантикой. Данные читаются в порядке записи и не являются долговременным содержимым файла.
Q03
Почему pipe — это не очередь сообщений?
→ Pipe хранит поток байт. Границы write() не обязаны совпадать с границами read(); приложение само восстанавливает framing.
Q04
Чем named pipe отличается от обычного pipe?
→ Named pipe имеет имя в файловой системе и может открываться неродственными процессами. Обычный pipe обычно передаётся через наследование FD после fork().
Q05
Named pipe хранит данные на диске?
→ Нет в обычном смысле. В ФС есть специальный объект-имя, но данные проходят через буфер ядра.
Q06
Почему signals не подходят для передачи больших данных?
→ Сигнал несёт очень мало информации, обычно идентификатор события. Это механизм уведомления, а не транспорт данных.
Q06a
Кто реально “доставляет” сигнал: процесс-отправитель или ядро?
→ Отправитель только делает syscall или порождает событие. Доставляет сигнал ядро: проверяет права, помечает pending, учитывает маску и при возврате целевого потока в userspace запускает handler/default action.
Q06b
Почему signal handler не является отдельным потоком?
→ Ядро меняет контекст существующего потока: кладёт signal frame на пользовательский стек и организует возврат в handler. После handler-а sigreturn восстанавливает старый контекст.
Q06c
Что происходит, если сигнал заблокирован?
→ Он остаётся pending до разблокировки. Исключения: SIGKILL и SIGSTOP нельзя заблокировать, проигнорировать или обработать.
Q06d
Почему в signal handler нельзя спокойно вызывать printf или malloc?
→ Handler асинхронно прерывает основной код. Если сигнал пришёл внутри malloc/printf, повторный вызов из handler-а может сломать внутреннее состояние или привести к deadlock. Нужны async-signal-safe действия.
Q07
В чём разница между SIGTERM и SIGKILL?
→ SIGTERM можно обработать и завершиться корректно. SIGKILL нельзя перехватить, игнорировать или обработать.
Q08
Что значит “message queue сохраняет границы сообщений”?
→ Отправленное сообщение остаётся отдельной единицей. Получатель забирает сообщение, а не произвольный кусок потока байт.
Q09
Где в Linux находятся POSIX message queues?
→ В отдельной виртуальной файловой системе /dev/mqueue.
Q10
Почему message queue не привязана к конкретному процессу?
→ Это объект ядра с именем. Процессы могут открывать его независимо, если имеют права.
Q11
В чём главное преимущество shared memory?
→ Высокая скорость для больших данных: процессы работают с одним общим регионом памяти, избегая постоянного копирования сообщений через ядро.
Q12
Почему shared memory опасна без синхронизации?
→ Несколько процессов могут одновременно читать/писать одну структуру, получая гонки, потерянные обновления и неконсистентные данные.
Q13
Какие системные вызовы нужны для POSIX shared memory?
→ Обычно shm_open() для объекта, ftruncate() для размера, mmap() для отображения в адресное пространство.
Q14
Если два процесса mmap'нули один shared memory объект, адреса у них обязаны совпадать?
→ Нет. Виртуальные адреса могут быть разными, но они указывают на один общий физический/ядровой объект.
Q15
Почему TCP/UDP-сокеты не лучший выбор для IPC на одной машине?
→ Они используют сетевые абстракции и имеют лишние накладные расходы. Для локального обмена часто лучше Unix domain sockets или shared memory.
Q16
Чем Unix domain socket отличается от TCP socket?
→ Интерфейс похож, но Unix domain socket работает только локально и передаёт данные через ядро без настоящей IP-коммуникации.
Q17
Какая особая возможность Unix domain sockets часто спрашивается на проверка понимания?
→ Передача файловых дескрипторов между процессами, например через SCM_RIGHTS.
Q18
Что такое named Unix domain socket?
→ Сокет, привязанный к пути в файловой системе, например /run/app.sock. Клиенты подключаются по этому пути.
Q19
Что такое abstract Unix domain socket?
→ Сокет, не привязанный ни к ФС, ни к конкретному процессу; имеет уникальное имя в network namespace.
Q20
Что выбрать для больших данных между локальными процессами?
→ Shared memory, но обязательно с отдельным механизмом синхронизации.
Q21
Что выбрать для простого shell pipeline?
→ Pipe: он идеально подходит для потока байт от stdout одного процесса к stdin другого.
Q22
Что выбрать для локального daemon/client протокола?
→ Часто Unix domain socket: socket API, локальная эффективность, права доступа, возможность передавать FD/credentials.
Q23
Почему “FIFO” в pipe не означает “очередь структурированных сообщений”?
→ FIFO относится к порядку байтов. Структуру сообщений поверх pipe нужно реализовать самому.
Q24
Какой IPC выбрать, если процессы должны общаться на разных машинах?
→ Сетевые сокеты: TCP, UDP или более высокоуровневый протокол поверх них.
Q25
Почему IPC почти всегда связан с компромиссом?
→ Удобство, скорость, сохранение границ сообщений, безопасность, синхронизация, локальность и возможность работы по сети редко достигаются одним механизмом одновременно.
