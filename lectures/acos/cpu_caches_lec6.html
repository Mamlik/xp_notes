⚡Мотивация: почему вообще нужны кеши

Процессор работает очень быстро. Типовая частота современного CPU — около 5 GHz. Это означает примерно:

1 такт ≈ 1 / (5 · 10⁹) секунды ≈ 0.2 ns

А доступ в оперативную память, то есть в DRAM, занимает примерно 60–120 ns. На первый взгляд это тоже очень мало, но для процессора это огромная задержка.

Главная проблема

Если CPU работает с тактом 0.2 ns, а RAM отвечает за 100 ns, то один поход в память может стоить примерно 500 тактов. Это как если бы процессор сделал запрос и очень долго простаивал, ничего полезного не делая.

Ключевая идея: кеши нужны не потому, что RAM «плохая», а потому что CPU слишком быстрый. Между скоростью процессора и скоростью памяти огромный разрыв, и кеши пытаются его спрятать.

Кеш — это маленькая, но очень быстрая память рядом с процессором. Она хранит копии тех участков RAM, к которым процессор недавно обращался или, скорее всего, скоро обратится.

Почему кеширование вообще работает?

Потому что у программ обычно есть локальность обращений:

Temporal locality Временная локальность

Если программа недавно использовала данные, велика вероятность, что она скоро использует их снова. Пример: переменная в цикле.

Spatial locality Пространственная локальность

Если программа обратилась к адресу X, велика вероятность, что скоро понадобится X+1, X+2 и соседние адреса. Пример: проход по массиву.

Важно понимать: кеш не хранит «переменные» как сущности языка C/C++. Он хранит куски памяти фиксированного размера — кеш-линии. Поэтому соседние переменные могут случайно оказаться в одной кеш-линии.

📦Кеш-линия: минимальная единица кеширования

Кеш-линия — минимальный кусок памяти, который кеш загружает и хранит целиком. Типичный размер кеш-линии — 64 байта.

Пример

Допустим, программа читает один байт по адресу 0x1000. Процессор обычно не загружает только этот один байт. Он загружает всю кеш-линию: например, адреса 0x1000..0x103F, то есть 64 байта.

Зачем грузить сразу 64 байта?

Из-за пространственной локальности. Если ты читаешь элемент массива a[i], скорее всего, скоро будешь читать a[i+1], a[i+2] и так далее. Они лежат рядом, значит уже окажутся в кеше.

Хороший случай Последовательный обход массива

for i = 0..n: соседние элементы лежат рядом. Один промах загружает целую линию, потом несколько обращений идут быстро.

Плохой случай Прыжки по памяти

Если программа каждый раз прыгает в далёкий участок памяти, почти каждое обращение может давать cache miss.

Ловушка: если ты используешь только 1 байт из каждой 64-байтной кеш-линии, то фактически тащишь из памяти много лишнего. Это плохо для производительности, потому что пропускная способность памяти расходуется впустую.
// 04

🧩Как адрес попадает в кеш: direct-mapped, fully associative, N-way

Кеш не может хранить всю RAM. Поэтому для каждого адреса нужно решить: в какое место кеша его можно положить. От этого зависит структура кеша.

1. Direct-mapped cache

Простой, но конфликтный Каждая кеш-линия из памяти может попасть ровно в одно место кеша

Плюс: очень просто и быстро искать. Минус: если два часто используемых адреса попадают в один и тот же слот кеша, они будут постоянно вытеснять друг друга.

Проблема direct-mapped: даже если кеш почти пустой, два «неудачных» адреса могут конфликтовать из-за одинакового index. Получается много промахов не потому, что кеш маленький, а потому что адреса мапятся в одно место.

2. Fully associative cache

Гибкий, но дорогой Любая кеш-линия может лежать в любом месте кеша

Плюс: почти нет конфликтов по месту. Минус: чтобы найти линию, нужно сравнить tag сразу со множеством мест. Это сложно, дорого и плохо масштабируется.

3. N-way set associative cache

Практический компромисс Кеш делится на sets, а внутри каждого set есть N мест — ways

Адрес выбирает конкретный set по index, но внутри этого set линия может лежать в одном из N ways. Это сильно уменьшает конфликты по сравнению с direct-mapped, но не требует поиска по всему кешу.

Пример 8-way set associative

Если кеш 8-way, то для каждого index есть 8 возможных слотов. Адрес по-прежнему попадает только в один set, но внутри него есть 8 вариантов, куда положить кеш-линию.

Тип кешаКуда можно положить линиюПлюсыМинусы
Direct-mapped Только в одно место Простой, быстрый, дешёвый Много conflict misses
Fully associative В любое место кеша Минимум конфликтов Сложный и дорогой поиск
N-way set associative В один set, но в любой из N ways Хороший баланс скорости и гибкости Конфликты всё ещё возможны
Важно понимать: современные кеши процессора обычно именно N-way set associative. Это «лучшее из двух миров»: не так конфликтно, как direct-mapped, и не так дорого, как fully associative.

🏷️Структура адреса: index, tag, offset

Чтобы понять, как кеш ищет данные, нужно разделить адрес на три части: offset, index и tag.

TAG какая именно область памяти
INDEX в какой set кеша идти
OFFSET байт внутри кеш-линии

Offset

Offset выбирает конкретный байт внутри кеш-линии. Если кеш-линия 64 байта, то нужно 6 бит, потому что 2⁶ = 64.

Index

Index выбирает set в кеше. Например, если в кеше 512 sets, нужно 9 бит, потому что 2⁹ = 512.

Tag

Tag нужен, чтобы проверить, действительно ли в выбранном set лежит нужная кеш-линия. Потому что много разных адресов могут иметь один и тот же index.

1
CPU получает адрес Например, программа читает переменную по некоторому адресу.
2
Offset говорит, какой байт нужен Но сначала надо найти всю кеш-линию.
3
Index выбирает set Процессор смотрит только в один set, а не во весь кеш.
4
Tag сравнивается с tag'ами ways Если найдено совпадение и линия валидна — cache hit. Если нет — cache miss.

PIPT и VIPT addressing

Современные программы работают с виртуальными адресами, а RAM и физические кеш-линии существуют по физическим адресам. Поэтому возникает вопрос: какими адресами индексировать и тегировать кеш?

PIPT Physically Indexed, Physically Tagged

Кеш индексируется и проверяется по физическому адресу. Это просто для корректности, но надо сначала перевести виртуальный адрес в физический через TLB/page table.

VIPT Virtually Indexed, Physically Tagged

Index берётся из виртуального адреса, а tag — из физического. Это ускоряет доступ: поиск в кеше можно начать параллельно с трансляцией адреса.

Почему VIPT работает аккуратно: обычно для L1 кеша index берётся из младших бит адреса, которые совпадают у виртуального и физического адреса внутри страницы. Поэтому можно начать доступ быстро, но всё равно проверить физический tag для корректности.
// 06

✂️Почему L1 делится на L1d и L1i

L1 кеш обычно разделён на две части:

L1i Instruction cache

Хранит инструкции программы — то, что CPU должен исполнять.

L1d Data cache

Хранит данные программы — переменные, элементы массивов, структуры и т. д.

Зачем такое разделение?

Причина №1: параллелизм

Процессору часто нужно одновременно загружать следующую инструкцию и читать/писать данные. Если инструкции и данные лежат в разных L1-кешах, эти операции проще выполнять параллельно.

Причина №2: разные паттерны доступа

Инструкции обычно читаются последовательно, а данные могут читаться и писаться хаотично. Разным типам обращений удобнее иметь отдельные структуры.

Причина №3: защита от вытеснения

Если бы инструкции и данные постоянно конкурировали за один маленький L1, активная работа с данными могла бы вытеснять инструкции, и наоборот.

Важно: разделение L1 на instruction/data — это пример Harvard-like подхода внутри CPU, хотя обычная память программы снаружи выглядит как единое адресное пространство.

📜Политики кеша: inclusion и write policies

Политики включения: inclusive, exclusive, non-inclusive

Когда есть несколько уровней кеша, возникает вопрос: должна ли линия, лежащая в L1, обязательно лежать ещё и в L2/L3?

ПолитикаИдеяПлюсыМинусы / ловушки
Inclusive Всё, что есть в L1, есть и в более нижнем уровне, например L3. Проще отслеживать, у какого ядра может быть линия. Удобнее для когерентности. Данные дублируются, эффективная ёмкость меньше. Вытеснение из L3 может заставить вытеснить из L1/L2.
Exclusive Линия лежит либо в L1, либо в L2/L3, но не дублируется. Больше эффективная ёмкость кешей. Сложнее управление: при перемещении между уровнями надо аккуратно переносить линии.
Non-inclusive Кеш не гарантирует ни строгого включения, ни строгого исключения. Гибкий компромисс. Сложнее рассуждать на пальцах; детали зависят от конкретной микроархитектуры.
Ловушка: inclusive не значит «лучше», а exclusive не значит «быстрее всегда». Это разные компромиссы между простотой когерентности, эффективным размером кеша и сложностью реализации.

Политики модификации: write-through и write-back

Когда процессор пишет в память, запись сначала попадает в кеш. Но когда она должна попасть в следующий уровень или в RAM?

Write-through Запись сразу проходит дальше

При записи данные обновляются и в кеше, и в следующем уровне памяти. Проще с точки зрения согласованности, но медленнее и создаёт больше трафика.

Write-back Запись откладывается

Сначала меняется только кеш-линия. В следующий уровень данные попадут позже — например, когда линия будет вытеснена. Быстрее, но требует отслеживать «грязные» линии.

Dirty cache line

В write-back кеше изменённая линия называется dirty, потому что её содержимое отличается от того, что лежит ниже по иерархии. Перед вытеснением dirty-линии её нужно записать обратно.

Связь с MESI: состояние Modified в MESI как раз означает, что линия изменена в этом кеше и отличается от памяти/нижнего уровня.

🔄Когерентность кешей: почему ядра должны договариваться

В многоядерном процессоре у каждого ядра есть свои L1/L2 кеши. Но память у программы логически одна. Значит, если одно ядро изменило переменную, другие ядра должны не продолжать бесконечно читать старую копию из своего кеша.

Проблема без когерентности

Ядро 1 читает переменную x = 0 и кеширует её. Ядро 2 записывает x = 1 в свою копию. Если ядро 1 продолжит читать старую кеш-линию, оно увидит 0, хотя логически значение уже изменилось.

Наивный протокол: инвалидация при записи

Простая идея: когда одно ядро хочет писать в кеш-линию, остальные копии этой линии у других ядер нужно сделать недействительными.

1
Несколько ядер читают одну линию Копия кеш-линии может быть у разных ядер.
2
Одно ядро хочет записать Перед записью оно должно получить эксклюзивное право владеть линией.
3
Остальные копии инвалидируются Другие ядра помечают свои копии как Invalid.
4
Запись выполняется Теперь только одно ядро имеет актуальную изменённую копию.

Алгоритм MESI

MESI — это протокол когерентности кешей. У каждой кеш-линии в каждом кеше есть одно из четырёх состояний:

M
Modified

Линия есть только в этом кеше, она изменена и отличается от памяти. При вытеснении её надо записать обратно.

E
Exclusive

Линия есть только в этом кеше, но она чистая: совпадает с памятью. Можно писать без рассылки invalidation.

S
Shared

Линия может быть в нескольких кешах одновременно. Читать можно, писать нельзя без получения эксклюзивности.

I
Invalid

Копия недействительна. Читать её нельзя: нужно заново получить линию из другого кеша/памяти.

Типичные переходы состояний

СитуацияЧто происходитИтоговое состояние
Ядро читает линию, которой нет у других Линия загружается в кеш, никто больше её не держит. Exclusive
Ядро читает линию, которая уже есть у других Линия становится общей для нескольких кешей. Shared
Ядро пишет в Exclusive Инвалидация не нужна: других копий нет. Modified
Ядро пишет в Shared Нужно инвалидировать копии у других ядер. Modified
Другое ядро читает Modified-линию Владелец должен отдать актуальные данные; состояние меняется. Shared / зависит от реализации
Линия вытесняется из Modified Перед удалением из кеша её надо записать в следующий уровень. Invalid после вытеснения

Зачем нужно состояние E? MESI vs MSI

В MSI есть только Modified, Shared, Invalid. Тогда даже если линия есть только у одного ядра, она часто сначала считается shared, и для записи может потребоваться лишняя операция получения эксклюзивности.

Состояние E — оптимизация: если ядро прочитало линию и никто другой её не держит, она попадает в Exclusive. Потом при записи она может перейти в Modified локально, без рассылки invalidation. Это уменьшает трафик между ядрами.

Модификации MESI: MOESI и MESIF

MOESI + Owned

Состояние Owned означает: линия изменена относительно памяти, но может быть shared между ядрами; один кеш отвечает за актуальную копию и запись обратно.

MESIF + Forward

Состояние Forward выбирает один кеш, который будет отвечать на запросы других ядер по shared-линии. Это снижает дублирование ответов.

Важно понимать: обычно не нужно рисовать всю state machine MESI, но важно понимать смысл состояний и почему запись в shared-линию вызывает инвалидации.

💥False sharing: когда разные переменные мешают друг другу

False sharing возникает, когда разные потоки работают с разными переменными, но эти переменные случайно лежат в одной кеш-линии.

Почему это проблема

Когерентность работает на уровне кеш-линий, а не отдельных переменных. Если два ядра пишут в разные переменные внутри одной 64-байтной линии, протокол MESI всё равно считает, что они конфликтуют за одну и ту же линию.

1
Переменные A и B лежат в одной кеш-линии Например, два счётчика в одной структуре.
2
Поток 1 пишет A, поток 2 пишет B Логически данные независимы.
3
Но железо видит одну кеш-линию Запись одного ядра инвалидирует копию всей линии у другого ядра.
4
Начинается cache line ping-pong Линия постоянно переезжает между ядрами, производительность резко падает.
Ловушка: false sharing — это не data race в смысле корректности программы. Потоки могут писать в разные переменные, программа может быть логически правильной, но работать очень медленно.

Как избежать false sharing?

  • Разносить горячие переменные по разным кеш-линиям, если они активно изменяются разными потоками.
  • Использовать padding/alignment, например выравнивание структуры до 64 байт.
  • Группировать данные по потоку: пусть каждый поток пишет в свой отдельный блок памяти.
  • Собирать результаты в конце, а не заставлять все потоки постоянно обновлять общий счётчик.
Мини-пример идеи

Плохо: массив счётчиков counter[thread_id], где соседние счётчики лежат рядом и попадают в одну кеш-линию. Лучше: сделать каждый счётчик отдельной cache-line-aligned структурой или добавить padding.

🛠️Практические советы для низкоуровневого кода

Понимание кешей полезно не только для экзамена. Оно помогает ускорять реальный код, особенно если код работает с большими массивами, структурами данных, многопоточностью и горячими циклами.

Совет 1 Следить за расположением данных

Горячие данные, которые используются вместе, лучше хранить рядом. Тогда один cache miss принесёт сразу несколько нужных значений.

Совет 2 Обходить память последовательно

Последовательный проход по массиву обычно намного дружелюбнее к кешу, чем случайные прыжки по памяти.

Совет 3 Осторожно со структурами

Иногда выгоднее Structure of Arrays вместо Array of Structures, если горячий код использует только часть полей.

Совет 4 Избегать false sharing

Если разные потоки часто пишут в разные переменные, проверь, не лежат ли они в одной кеш-линии.

Array of Structures vs Structure of Arrays

AoS: Array of Structures

struct Point { float x, y, z; } points[N];

Удобно, когда часто нужны все поля одной сущности: x, y, z вместе.

SoA: Structure of Arrays

float x[N], y[N], z[N];

Лучше, когда горячий код проходит только по одному полю, например только по x. Тогда кеш не тащит лишние y и z.

Главная практическая мысль: быстрый код — это не только меньше операций. Иногда важнее, чтобы данные приходили из L1/L2, а не из DRAM, и чтобы ядра не дрались за одну кеш-линию.

✅Что обязательно запомнить

  • CPU быстрый, DRAM медленная: 1 такт около 0.2 ns, доступ к RAM — десятки/сотни ns.
  • Кеш-линия — минимальная единица кеширования, обычно 64 байта.
  • Современные кеши обычно N-way set associative.
  • Адрес делится на tag/index/offset: offset выбирает байт, index — set, tag проверяет, та ли линия.
  • L1 делится на L1d и L1i, чтобы данные и инструкции не мешали друг другу и могли грузиться параллельно.
  • Write-back быстрее, но требует dirty state; write-through проще, но создаёт больше трафика.
  • MESI поддерживает когерентность кешей между ядрами.
  • Exclusive в MESI нужно, чтобы писать в единственную чистую копию без лишних инвалидаций.
  • False sharing — разные переменные в одной кеш-линии тормозят многопоточный код.

🎯Вопросы на понимание

Q01
Почему доступ к DRAM считается медленным, если 100 ns — это очень мало?
→ Для человека мало, для CPU много. При 5 GHz один такт ≈ 0.2 ns, значит 100 ns — это около 500 тактов ожидания.
Q02
Почему кеш загружает 64 байта, если программа запросила только 4 байта?
→ Кеш работает кеш-линиями. Это использует spatial locality: скорее всего, скоро понадобятся соседние байты.
Q03
Чем cache hit отличается от cache miss?
→ Hit: нужная кеш-линия уже есть на нужном уровне кеша. Miss: её нет, нужно идти на следующий уровень — L2/L3/DRAM, что дороже.
Q04
Почему direct-mapped cache может плохо работать даже при большом размере кеша?
→ Потому что каждый адрес может попасть только в один слот. Два часто используемых адреса с одинаковым index будут постоянно вытеснять друг друга.
Q05
Почему fully associative cache не делают огромным и основным?
→ Нужно сравнивать tag с большим числом возможных мест. Это дорого по железу, энергии и задержке.
Q06
Почему N-way set associative cache называют компромиссом?
→ Адрес попадает в конкретный set, но внутри set есть N вариантов. Меньше конфликтов, чем в direct-mapped, но проще, чем fully associative.
Q07
Что делают tag, index и offset?
→ Offset выбирает байт внутри кеш-линии, index выбирает set, tag проверяет, что в выбранном set лежит именно нужная линия.
Q08
Если кеш-линия 64 байта, сколько бит offset нужно?
→ 6 бит, потому что 2⁶ = 64.
Q09
В чём разница между PIPT и VIPT?
→ PIPT: и index, и tag по физическому адресу. VIPT: index по виртуальному, tag по физическому; это позволяет начать доступ к L1 параллельно с трансляцией адреса.
Q10
Зачем L1 кеш делят на L1i и L1d?
→ Чтобы инструкции и данные можно было загружать параллельно, чтобы они не вытесняли друг друга и потому что у них разные паттерны доступа.
Q11
Чем inclusive cache отличается от exclusive cache?
→ Inclusive: если линия есть в L1, она есть и в нижнем уровне. Exclusive: линия старается не дублироваться между уровнями, что увеличивает эффективную ёмкость.
Q12
Почему write-back обычно быстрее write-through?
→ Write-back не пишет каждое изменение сразу дальше по иерархии. Он откладывает запись до вытеснения dirty-линии, уменьшая трафик.
Q13
Что такое dirty cache line?
→ Кеш-линия, изменённая в кеше и ещё не записанная в следующий уровень/память. В MESI это соответствует состоянию Modified.
Q14
Зачем вообще нужна когерентность кешей?
→ Чтобы разные ядра не работали бесконечно со старыми копиями одной и той же памяти. Если одно ядро изменило линию, другие должны узнать, что их копии устарели.
Q15
Что означают состояния MESI?
→ Modified: изменена и только тут. Exclusive: только тут, но чистая. Shared: есть у нескольких, чистая. Invalid: копия недействительна.
Q16
Зачем в MESI состояние Exclusive, если есть Shared и Modified?
→ Exclusive позволяет при первой записи перейти в Modified без рассылки invalidation, если известно, что других копий нет.
Q17
Что происходит, если ядро хочет писать в Shared-линию?
→ Оно должно получить эксклюзивное владение: отправить invalidation другим ядрам, чтобы их копии стали Invalid, и затем писать.
Q18
В чём идея MOESI и состояния Owned?
→ Owned позволяет иметь shared-копии линии, которая при этом модифицирована относительно памяти; один кеш является владельцем актуальной версии.
Q19
Что такое false sharing и почему это не обязательно ошибка корректности?
→ Потоки могут писать в разные переменные, но если они лежат в одной кеш-линии, железо считает, что конфликт идёт за всю линию. Программа корректна, но медленная.
Q20
Как практически бороться с false sharing?
→ Разносить горячие переменные по разным кеш-линиям, использовать padding/alignment, хранить per-thread данные отдельно и объединять результаты позже.
Q21
Почему последовательный обход массива обычно быстрее случайного обхода?
→ Последовательный обход использует spatial locality и prefetching: одна кеш-линия приносит несколько соседних элементов. Случайный обход часто даёт miss на каждом шаге.
Q22
Когда Structure of Arrays может быть лучше Array of Structures?
→ Когда горячий код использует только одно поле из многих. SoA позволяет не тащить в кеш ненужные поля.
