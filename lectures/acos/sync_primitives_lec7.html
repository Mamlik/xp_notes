🧭Карта темы: что такое синхронизация

Синхронизация нужна, когда несколько потоков, процессов или ядер могут одновременно работать с общими данными. Без синхронизации возникает race condition: результат зависит от случайного порядка выполнения.

Главная идея lock-а

Lock выделяет критическую секцию — участок кода, где поток работает с общим состоянием и должен быть один.

lock → critical section → unlock

Пока один поток держит lock, остальные либо крутятся, либо спят, либо пытаются использовать гибридный подход.

Проблема 1 Гонки данных

Два ядра одновременно меняют одну переменную: одно изменение может потеряться.

Проблема 2 Кеши и атомики

Lock часто заставляет ядра перекидывать одну кеш-линию туда-сюда.

Проблема 3 Системные вызовы

Сон и пробуждение требуют помощи ОС, а переход в ядро стоит дорого.

Ключевой вопрос темы: что дешевле в конкретной ситуации — немного покрутиться на CPU или уйти спать через ОС?
// 01

🔁Spinlock: крутимся, пока lock не освободится

Spinlock — самый простой lock. Если lock занят, поток не засыпает, а продолжает в цикле проверять, можно ли его захватить. Такой цикл называется busy loop или busy waiting.

// Упрощённая идея spinlock
while (atomic_exchange(&lock, 1) == 1) {
    // lock занят — крутимся
}

// critical section

atomic_store(&lock, 0);
1
Поток пытается атомарно поставить lock = 1 Операция должна быть атомарной, иначе два потока могут одновременно “увидеть”, что lock свободен.
2
Если получилось — входим в критическую секцию Теперь текущий поток владеет lock-ом.
3
Если не получилось — повторяем попытку Поток не отдаёт CPU, а постоянно проверяет lock.

Когда spinlock хорош

Критическая секция очень короткая. Например, несколько инструкций в ядре.

Нельзя спать. Например, код выполняется в контексте, где планировщик недоступен.

Ожидание меньше цены системного вызова. Тогда выгоднее покрутиться.

Когда spinlock плох

Критическая секция длинная. Поток сжигает CPU впустую.

Держатель lock-а вытеснен. Остальные крутятся, а тот, кто должен освободить lock, вообще не работает.

Много ядер спорят за один lock. Начинается сильный cache-line bouncing.

Почему busy loop плох: он тратит CPU-время, мешает другим потокам, гоняет кеш-линию между ядрами, нагревает процессор и повышает энергопотребление. На мобильных устройствах это особенно важно: чем выше активность CPU, тем быстрее садится батарея.
// 02

⚙️Почему busy loop + atomic write — ужасно

Наивный spinlock часто делают так: каждый ожидающий поток постоянно выполняет атомарную операцию записи, пытаясь захватить lock. Это очень дорого.

// Плохо: каждая итерация делает atomic write / exchange
while (atomic_exchange(&lock, 1) == 1) {
    // каждый раз пытаемся записать в lock
}
Почему atomic write бьёт по производительности

Lock обычно лежит в одной кеш-линии. Чтобы ядро могло записать в эту кеш-линию, оно должно получить её в эксклюзивное/modified состояние. Если много ядер постоянно пишут в одну и ту же линию, она всё время переезжает между кешами разных ядер.

Итог: busy loop с постоянной атомарной записью создаёт огромный трафик когерентности кешей. Потоки не просто ждут — они активно мешают друг другу.

Улучшение: busy loop + atomic read

Менее ужасный вариант: сначала читать lock обычной/атомарной загрузкой, а дорогую атомарную запись делать только когда кажется, что lock свободен.

// Идея test-and-test-and-set
while (true) {
    while (atomic_load(&lock) == 1) {
        // только читаем, не пишем
    }
    if (atomic_exchange(&lock, 1) == 0) {
        break; // захватили
    }
}

busy loop + atomic write

Каждая попытка ожидания — запись.

Постоянная инвалидация кеш-линий.

Очень плохо при конкуренции.

busy loop + atomic read

Пока lock занят, в основном идут чтения.

Кеш-линия может быть shared у многих ядер.

Дорогая запись только при попытке захвата.

Но: даже atomic read loop всё равно сжигает CPU. Это не универсальное решение, а просто менее плохой spinlock.
// 03

😴Mutex: если не можешь захватить — спи

Mutex решает главную проблему spinlock-а: поток не обязан крутиться, если lock занят. Он может отдать CPU другому процессу или треду.

Интуиция

Если критическая секция длинная, ждать в busy loop бессмысленно. Лучше сказать ОС: “разбуди меня, когда lock освободится”, и дать процессору заняться полезной работой.

1
Поток пытается захватить mutex Если mutex свободен — захват быстрый.
2
Если mutex занят — поток уходит спать Нужна поддержка ОС: системный вызов переводит поток в состояние ожидания.
3
Владелец освобождает mutex ОС будит одного или нескольких ожидающих.
4
Проснувшийся поток снова конкурирует за lock Пробуждение не всегда означает мгновенный захват.
Ключевая цена mutex-а: если приходится спать/будить, нужны системные вызовы и работа планировщика. Это дороже, чем несколько итераций spinlock-а, но дешевле, чем долго жечь CPU.

Mutex в xv6

В учебной ОС xv6 идея mutex-а хорошо показывает роль ядра: нужен один механизм, который усыпит текущий поток/процесс, и парный механизм, который его разбудит. Без ОС поток не может сам “исчезнуть” с CPU корректно и проснуться по событию.

sleep Отправить процесс спать

Процесс перестаёт занимать CPU и ждёт события.

wakeup Разбудить ожидающих

Когда lock освобождён, ядро может вернуть ожидающий процесс в очередь готовых.

// 04

🧪Futex: быстрый путь в userspace, сон — только если надо

Futex расшифровывают как fast userspace mutex. Идея: не ходить в ядро, если lock можно захватить быстро в пользовательском пространстве.

Почему futex вообще появился

Системные вызовы дорогие. Но чистый spinlock тоже плох, если ждать приходится долго. Futex соединяет два подхода: сначала быстрые атомики в userspace, а если не получилось — системный вызов, чтобы заснуть.

1
Пробуем захватить lock атомарно в userspace Если lock свободен, ядро вообще не участвует.
2
Если lock занят — можно немного покрутиться Это выгодно, если владелец скоро освободит lock.
3
Если не вышло — futex_wait Поток просит ядро усыпить его, но только если значение futex-слова всё ещё ожидаемое.
4
При unlock — futex_wake Освобождающий поток будит ожидающих через ядро, если они есть. 
// Упрощённая логика futex-lock
if (atomic_try_lock(&lock)) {
    return; // fast path: без syscall
}

for (int i = 0; i < SPIN_LIMIT; i++) {
    if (atomic_try_lock(&lock)) return;
}

futex_wait(&lock, LOCKED); // slow path: syscall, сон
Главное: futex нормален именно потому, что uncontended case не требует syscall. Если lock свободен, всё происходит через обычные атомарные операции в userspace.
Ловушка: futex — это не “готовый mutex”, а низкоуровневый механизм ядра для сна/пробуждения по адресу в памяти. Из него уже строят mutex, condition variable и другие примитивы.
// 05

📚RWLock: много читателей или один писатель

RWLock — read-write lock. Он полезен, когда есть данные, которые часто читают и редко изменяют.

rlock / runlock Режим чтения

Несколько читателей могут держать lock одновременно, потому что они не меняют данные.

wlock / wunlock Режим записи

Писатель должен быть один. Пока пишет один поток, читатели и другие писатели ждут.

Много читателей одновременно OR один писатель эксклюзивно

Зачем настраивать приоритет

В зависимости от workload-а RWLock может быть настроен по-разному:

Приоритет читателей

Новые читатели могут проходить, даже если писатель уже ждёт.

Плюс: высокая пропускная способность чтений.

Минус: писатель может голодать, если читатели идут непрерывно.

Приоритет писателей

Если писатель ждёт, новых читателей могут блокировать.

Плюс: писатель не ждёт бесконечно.

Минус: падает параллелизм чтений.

Ловушка: RWLock не всегда быстрее mutex-а. Если записей много или критическая секция короткая, накладные расходы RWLock-а могут съесть всю пользу.
// 06

🧷CPU local lock: локальная защита внутри ядра

CPU local lock — особый тип примитива, который используется в ядре. Его смысл отличается от обычного mutex-а или spinlock-а.

Что значит local

Local означает, что lock защищает не от всех ядер сразу, а от асинхронных обработчиков на текущем CPU: например, interrupt handler, softirq, preemption-зависимых ситуаций.

Только ядро Не userspace-примитив

Обычные программы не используют CPU local lock напрямую.

Асинхронность Защита от обработчиков

Код может быть прерван обработчиком, который трогает те же per-CPU данные.

Локальность Один CPU

Главная цель — выделить участок, где локальные асинхронные события не вмешаются.

Интуиция: обычный lock отвечает на вопрос “что если другой поток на другом ядре?”. CPU local lock отвечает на вопрос “что если меня прямо сейчас прервёт локальный обработчик?”.
// 07

⚖️Достоинства и недостатки подходов

Выбор примитива синхронизации — это всегда компромисс между задержкой, расходом CPU, числом системных вызовов, справедливостью и поведением кешей.

Подход Что делает ожидающий поток Плюсы Минусы Когда выбирать
Spinlock Крутится в busy loop Нет syscall, очень быстрый при коротком ожидании Жжёт CPU, cache-line bouncing, плохо при длинном ожидании Очень короткие критические секции, kernel-код, нельзя спать
Mutex Уходит спать через ОС Не тратит CPU во время ожидания Syscall, планировщик, context switch Длинные критические секции, ожидание может быть долгим
Futex Сначала userspace, потом сон Fast path без syscall, slow path через ядро Сложная реализация, надо аккуратно избегать lost wakeup Практические mutex-ы в userspace
RWLock Читатели делят доступ, писатели ждут эксклюзивности Хорош для read-mostly структур Сложность, starvation, накладные расходы Много чтений, мало записей, чтения достаточно длинные
CPU local lock Ограничивает локальную асинхронность Подходит для per-CPU структур в ядре Не заменяет глобальную синхронизацию между ядрами Kernel-код, защита от локальных interrupt/softirq/preemption ситуаций

О чём думать при выборе lock-а

  • Большая ли критическая секция? Чем она длиннее, тем хуже spinlock.
  • Заставляем ли ядра перекидывать кеш-линии? Частый atomic write по одному адресу плохо масштабируется.
  • Делаем ли лишние системные вызовы? Если lock почти всегда свободен, syscall на каждый lock/unlock — лишняя цена.
  • Можно ли избежать syscall-ов? Futex старается делать быстрый путь без ядра.
  • Есть ли гарантия, что процесс захватит lock? Некоторые lock-и могут приводить к starvation.
  • Честно ли захватывается lock? Fair lock обычно предсказуемее, но может быть медленнее.
// 08

🎛️Fairness, starvation и гарантии прогресса

Lock может быть быстрым, но “нечестным”. Это значит, что один поток может долго ждать, пока другие снова и снова успевают захватить lock раньше него.

Starvation Голодание

Поток формально может захватить lock, но на практике очень долго не получает шанс.

Fair lock Честный lock

Обычно обслуживает ожидающих в порядке очереди или близко к нему.

Unfair lock Нечестный lock

Может быть быстрее из-за меньших накладных расходов, но хуже по предсказуемости.

Компромисс: честность часто стоит производительности. Очередной lock может меньше гонять потоки хаотично, но требует дополнительных структур данных и атомарных операций.

Пример: почему “разбудили” не значит “захватил”

Если поток спал на mutex-е/futex-е и его разбудили, он ещё должен получить CPU и снова выполнить попытку захвата. За это время другой поток может успеть забрать lock.

Вывод: после пробуждения условие всегда проверяют заново. Нельзя считать, что wakeup автоматически передал владение lock-ом.
// 09

🧨Типичные баги с синхронизацией

Deadlock Взаимная блокировка

Поток A держит lock1 и ждёт lock2, поток B держит lock2 и ждёт lock1. Никто не продолжит выполнение.

Lost wakeup Потерянное пробуждение

Поток собирался заснуть, но событие произошло прямо перед сном. Если не проверить состояние атомарно, поток может уснуть навсегда.

Priority inversion Инверсия приоритетов

Низкоприоритетный поток держит lock, высокоприоритетный ждёт, а среднеприоритетные мешают низкому освободить lock.

Convoying Эффект колонны

Много потоков выстраиваются за одним lock-ом, и производительность падает из-за пробуждений и переключений контекста.

Главное правило: синхронизация должна защищать не “кусок кода ради куска кода”, а конкретный инвариант данных. Если непонятно, какой инвариант защищается, lock почти наверняка используется неправильно.
// 10

🧾Мини-шпаргалка

Что обязательно помнить
  • Spinlock — busy loop, хорош только при очень коротком ожидании или когда нельзя спать.
  • Busy loop + atomic write — плохо из-за постоянных записей и перекидывания кеш-линии между ядрами.
  • Busy loop + atomic read — лучше, потому что ожидание в основном читает shared кеш-линию.
  • Mutex — при занятости lock-а поток спит, нужна поддержка ОС: sleep/wakeup.
  • Futex — fast path в userspace без syscall, slow path через futex_wait/futex_wake.
  • RWLock — много читателей или один писатель; полезен при read-mostly нагрузке.
  • CPU local lock — kernel-only примитив для защиты от локальных асинхронных обработчиков на CPU.
  • Fairness — честность захвата; повышает предсказуемость, но может стоить скорости.
  • Starvation — поток долго не получает lock, хотя система в целом работает.
  • Выбор lock-а зависит от длины критической секции, конкуренции, syscall-ов, кешей и требований к честности.

🎯Вопросы на понимание

Q01
Почему spinlock не отправляет поток спать?
→ Потому что его идея — ждать активным циклом. Это выгодно, если ожидание очень короткое или если в данном контексте нельзя вызывать планировщик.
Q02
Почему spinlock плох для длинной критической секции?
→ Ожидающие потоки всё время занимают CPU, хотя не делают полезной работы. Если владелец lock-а долго не освобождает его, busy waiting превращается в чистую потерю ресурсов.
Q03
Почему busy loop + atomic write хуже, чем busy loop + atomic read?
→ Atomic write требует эксклюзивного владения кеш-линией и инвалидирует копии у других ядер. Atomic read может использовать shared состояние кеш-линии и создаёт меньше трафика когерентности.
Q04
Что такое test-and-test-and-set?
→ Сначала в цикле читаем lock, пока он занят, и только когда он выглядит свободным, делаем дорогую атомарную попытку записи/exchange.
Q05
Почему высокая частота CPU делает busy loop особенно неприятным?
→ Чем активнее CPU крутится, тем больше энергии он потребляет и тем сильнее греется. На мобильных устройствах это напрямую бьёт по батарее.
Q06
Почему mutex требует поддержки ОС?
→ Чтобы поток перестал выполняться и не занимал CPU, его должен усыпить планировщик ядра. Потом ядро должно разбудить его при освобождении lock-а.
Q07
Почему mutex может быть медленнее spinlock-а при очень коротком ожидании?
→ Сон и пробуждение требуют системных вызовов, работы планировщика и, возможно, переключения контекста. Это может быть дороже, чем несколько итераций spin.
Q08
Что такое futex и почему он быстрее обычного “всегда syscall” mutex-а?
→ Futex делает fast path в userspace: если lock свободен, системный вызов не нужен. В ядро идём только при реальном ожидании.
Q09
Почему futex_wait должен проверять значение слова перед сном?
→ Чтобы избежать lost wakeup: если lock уже освободился до сна, поток не должен засыпать по старому состоянию.
Q10
Почему futex — это не сам mutex?
→ Futex — низкоуровневый механизм wait/wake по адресу в памяти. Политика mutex-а строится поверх него в библиотеке/рантайме.
Q11
Когда RWLock лучше обычного mutex-а?
→ Когда данные читаются часто, пишутся редко, а чтения достаточно долгие, чтобы параллелизм читателей окупал накладные расходы RWLock-а.
Q12
Почему RWLock может оказаться хуже mutex-а?
→ Если записей много, критические секции короткие или высокая конкуренция, накладные расходы и блокировки писателей/читателей могут сделать RWLock медленнее.
Q13
Что значит “приоритет читателей” в RWLock-е и чем это опасно?
→ Новые читатели могут проходить, пока писатель ждёт. Это повышает throughput чтений, но писатель может голодать.
Q14
Что значит “приоритет писателей” в RWLock-е?
→ Если писатель ждёт, новых читателей могут не пускать, чтобы писатель не голодал. Минус — меньше параллелизма чтений.
Q15
Что защищает CPU local lock?
→ Он защищает критические секции ядра от локальных асинхронных обработчиков на текущем CPU, а не является обычным межпоточным userspace-lock-ом.
Q16
Почему CPU local lock не заменяет глобальный lock между ядрами?
→ Он локален для CPU. Если данные реально общие между ядрами, нужна синхронизация, которая учитывает параллельный доступ с других CPU.
Q17
Что такое cache-line bouncing в контексте lock-ов?
→ Кеш-линия с lock-ом постоянно переходит между кешами ядер, потому что разные ядра пытаются читать/писать её, особенно при atomic write.
Q18
Почему “разбудили поток” не значит “он уже владеет lock-ом”?
→ После wakeup поток только становится runnable. Ему ещё надо получить CPU и снова попытаться захватить lock; другой поток может опередить его.
Q19
Что такое fair lock?
→ Lock, который старается выдавать доступ ожидающим потокам в справедливом порядке, например FIFO. Это снижает starvation, но может стоить производительности.
Q20
Почему unfair lock иногда быстрее fair lock-а?
→ Ему не нужно строго поддерживать очередь и дополнительные гарантии. Он может позволить только что работавшему потоку быстро перезахватить lock, что иногда лучше для кешей.
Q21
Какие вопросы надо задать себе при выборе примитива синхронизации?
→ Длина критической секции, вероятность конкуренции, цена syscall-ов, cache-line bouncing, возможность сна, требования к справедливости и гарантии прогресса.
Q22
Почему нельзя всегда использовать самый “умный” lock?
→ Более сложный lock имеет свои накладные расходы: очереди, атомики, syscall-ы, дополнительные состояния. На простых сценариях он может быть хуже простого spinlock-а или mutex-а.
Q23
Что такое deadlock?
→ Ситуация, когда потоки ждут друг друга по циклу: A держит lock1 и ждёт lock2, B держит lock2 и ждёт lock1.
Q24
Что такое lost wakeup?
→ Пробуждение произошло до того, как поток реально заснул. Если состояние не проверяется атомарно, поток может уснуть и уже никогда не проснуться.
Q25
Почему критическая секция должна быть минимальной?
→ Чем дольше держится lock, тем больше ожидание, выше конкуренция, больше риск convoying и хуже масштабируемость.
Q26
Почему lock может ухудшить масштабируемость на многоядерной системе?
→ Все ядра начинают конкурировать за одну точку сериализации. Плюс возникает трафик когерентности кешей вокруг кеш-линии с lock-ом.
Q27
Когда loop + sleep приемлемее чистого busy loop?
→ Когда ожидание может быть заметным. Небольшой spin может поймать быстрый unlock, а sleep не даст потоку бесконечно тратить CPU.
Q28
Что важнее: атомарность операции или правильная политика ожидания?
→ Нужно и то и другое. Атомарность защищает корректность захвата, а политика ожидания определяет производительность и поведение под нагрузкой.
Q29
Почему lock — это не только про корректность, но и про производительность?
→ Разные lock-и по-разному влияют на CPU, кеши, syscall-и, планировщик, справедливость и энергопотребление.
Q30
Какой главный принцип выбора между spinlock и mutex?
→ Если ждать очень недолго и нельзя/невыгодно спать — spinlock. Если ожидание может быть длинным — mutex/futex, чтобы не жечь CPU.
