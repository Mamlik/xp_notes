01

Мотивация

Зачем вообще понимать компиляцию, линковку и ELF

Когда мы говорим «скомпилировать программу», в реальности обычно имеем в виду не одно действие, а цепочку преобразований: исходный код нужно обработать препроцессором, превратить в ассемблер, собрать в объектные файлы, связать с другими объектными файлами и библиотеками, а затем корректно загрузить в память при запуске.

Понимание этой цепочки помогает осознанно выбирать между разными способами сборки и понимать, почему программа иногда собирается долго, почему бинарь большой, почему возникают ошибки линковки, почему одни бинарники зависят от системы, а другие можно переносить почти куда угодно.

Удобство кода

Как писать и организовывать программу

Можно разбивать код на translation units, использовать заголовки, библиотеки, шаблоны, inline-функции. Но удобство часто влияет на скорость сборки и размер бинаря.

Скорость компиляции

Как быстро получить бинарь

Много include-файлов, шаблонов и тяжёлых оптимизаций увеличивают время сборки. Инкрементальная сборка пытается пересобирать только изменённые части.

Оптимизации

Насколько агрессивно менять код

-O0 удобнее для отладки, -O2/-O3 обычно быстрее, LTO позволяет оптимизировать между объектными файлами, но усложняет сборку.

Размер и переносимость

Статически или динамически

Статическая линковка делает бинарь более самостоятельным, но крупнее. Динамическая экономит память и диск, но зависит от библиотек в системе.

🎯
Главная идеяКомпиляция и линковка — это не «магия компилятора», а набор этапов, каждый из которых создаёт конкретный артефакт и решает конкретную задачу.
02

Из C/C++ кода в исполняемый файл

Четыре классических этапа: preprocessing, compilation, assembly, linking

1
PreprocessingПрепроцессор раскрывает #include, макросы #define, условную компиляцию #if/#ifdef. На выходе получается расширенный исходный код.
2
CompilationКомпилятор переводит расширенный C/C++ код в ассемблер конкретной архитектуры. Здесь происходят парсинг, type checking, оптимизации, генерация машинно-зависимого представления.
3
AssemblyАссемблер превращает ассемблерный текст в объектный файл: машинный код уже есть, но адреса внешних символов ещё не окончательны.
4
LinkingЛинкер объединяет несколько объектных файлов и библиотек, разрешает символы, применяет релокации и строит исполняемый файл или shared object.
Можно посмотреть этапы рукамиshell
gcc -E main.c -o main.i      # только preprocessing
gcc -S main.i -o main.s      # C -> assembly
gcc -c main.s -o main.o      # assembly -> object file
gcc main.o utils.o -o app    # linking -> executable
⚠️
Важная детальКоманда gcc main.c -o app скрывает все эти стадии под одной командой. Поэтому слово «компиляция» в быту часто означает весь pipeline целиком, хотя технически compilation — только один из этапов.
03

Translation units, символы и объектные файлы

Почему линковка вообще нужна

Translation unit — это исходный файл после раскрытия всех #include. Например, main.c вместе со всем текстом подключённых заголовков превращается в один большой поток кода, который компилятор обрабатывает независимо от других .c/.cpp файлов.

Проблема в том, что один translation unit может ссылаться на функции и переменные, определённые в другом translation unit. На этапе компиляции main.c компилятор ещё не обязан знать окончательный адрес внешней функции.

symbol definition

Определение символа

Файл реально содержит тело функции или место хранения глобальной переменной. Например, int foo() { ... } создаёт символ foo.

symbol reference

Ссылка на символ

Файл использует символ, но не знает его адрес. Например, вызов foo() из другого файла создаёт unresolved reference, которую должен заполнить линкер.

🔗
Роль линкераЛинкер собирает вместе объектные файлы, находит для каждой ссылки подходящее определение символа и исправляет машинный код/данные так, чтобы они ссылались на правильные адреса.
04

ELF: Executable and Linkable Format

Один формат для объектных файлов, исполняемых файлов и shared libraries

ELF — стандартный формат файлов в Linux/Unix-подобных системах для исполняемых программ, объектных файлов .o, shared libraries .so и core dumps.

ELF одновременно обслуживает две разные задачи: линковку и загрузку в память. Для линкера важны секции, символы и релокации. Для загрузчика важны сегменты и program headers.

🧩 Link-time view

Взгляд линкера

  • Секции: .text, .data, .bss, .symtab.
  • Таблицы символов и строк.
  • Релокации.
  • Объединение объектных файлов.
🚀 Load-time view

Взгляд ядра и загрузчика

  • Program headers.
  • Сегменты LOAD, DYNAMIC, INTERP.
  • Права страниц: read/write/execute.
  • Маппинг файла в виртуальную память.
✅
Ключевая формулировкаСекции — логическое представление для линковки. Сегменты — представление для загрузки в память при исполнении.
05

Из чего состоит ELF-файл

ELF header, section headers, program headers и сами данные

ELF header

Главный заголовок

Содержит ELF magic, класс 32/64-bit, endianness, архитектуру, тип файла, entry point, смещения до таблиц section/program headers.

Section header table

Таблица секций

Описывает секции: имя, тип, flags, offset в файле, размер, alignment, связь с другими секциями. Нужна в первую очередь линкеру и инструментам анализа.

Program header table

Таблица сегментов

Описывает, какие части файла нужно загрузить в память, по каким адресам, с какими правами и каким выравниванием. Используется загрузчиком.

Sections data

Содержимое секций

Код, данные, строки, символы, релокации, debug-информация, metadata для динамического линковщика.

09

Релокации в статической линковке

Как объектные файлы ссылаются друг на друга до знания финальных адресов

При компиляции объектный файл может ссылаться на символы из других translation units. Но на этом этапе адреса ещё неизвестны. Поэтому ассемблер оставляет в объектном файле не только машинный код, но и relocation entries: инструкции для будущего линкера, где и как нужно исправить адрес.

1
Компилятор видит внешний символНапример, main.o вызывает foo, но foo определена в foo.o.
2
В объектный файл пишется заглушкаВ машинном коде оставляется место, которое позже нужно заполнить адресом или смещением.
3
Создаётся relocation entryЗапись говорит: по такому offset нужно применить такую формулу к такому символу.
4
Линкер объединяет объектные файлыОн выбирает финальные адреса секций, находит определения символов и патчит нужные места.
🧮
Идея формулыРелокация обычно вычисляет значение вроде S + A - P, где S — адрес символа, A — addend, а P — адрес места, куда записывается релокация. Конкретная формула зависит от типа релокации и архитектуры.
REL

Неявный addend

Addend хранится прямо в месте, которое будет патчиться. Relocation entry не содержит отдельного поля addend.

RELA

Явный addend

Addend хранится явно в relocation entry. Это удобно для анализа и часто используется на x86-64.

RELR

Relative relocations compactly

Компактный формат для множества relative relocations, полезный для PIE/shared objects с большим числом относительных исправлений.

10

PIC, PIE и ASLR

Почему адреса не всегда можно знать заранее

PIC

Position Independent Code

Код, который может исполняться независимо от того, куда он загружен в память. Критически важен для shared libraries.

PIE

Position Independent Executable

Исполняемый файл, собранный как position-independent. Его базовый адрес может рандомизироваться при запуске.

ASLR

Address Space Layout Randomization

Механизм безопасности: стек, heap, shared libraries и PIE-бинарь загружаются по случайным адресам.

Если программа не position-independent, линкер может предполагать фиксированные адреса. Но с PIE/ASLR финальный адрес загрузки неизвестен до запуска. Поэтому часть адресов нельзя полностью вычислить на этапе статической линковки.

Такие релокации заполняются динамическим линковщиком при запуске. Для статических PIE-бинарей аналогичную работу может делать startup-код/C runtime.

🔐
Связь с безопасностьюPIE делает рандомизацию адреса основного исполняемого файла возможной. Без PIE ASLR всё ещё может рандомизировать библиотеки, стек и heap, но сам executable часто остаётся по фиксированному адресу.
11

Динамическая линковка

Зачем нужны shared objects и что делает ld.so

Многие программы используют один и тот же код: например, почти каждая C-программа использует libc.so. Если бы каждая программа включала в себя полную копию libc, это тратило бы место на диске и память в RAM.

Shared object (.so) позволяет хранить общую библиотеку отдельно и маппить её код в адресные пространства разных процессов. Кодовые read-only страницы библиотеки могут физически разделяться между процессами.

📦 Статическая линковка
  • Код библиотек попадает прямо в бинарь.
  • Меньше зависимость от installed libraries.
  • Бинарь крупнее.
  • Обновление библиотеки требует пересобрать бинарь.
🔗 Динамическая линковка
  • Библиотеки загружаются при запуске.
  • Код shared library можно делить между процессами.
  • Бинарь меньше.
  • Есть зависимость от ABI и версий библиотек в системе.
1
Ядро читает ELF executableВидит сегмент INTERP, который указывает путь до динамического линковщика.
2
Маппится dynamic linkerНапример, ld-linux-x86-64.so.2. Именно он готовит shared libraries.
3
ld.so читает DYNAMICНаходит DT_NEEDED, relocation tables, GOT/PLT, init/fini hooks и другую metadata.
4
Библиотеки маппятся в памятьДинамический линковщик находит нужные .so, загружает их сегменты, применяет нужные релокации.
5
Управление передаётся программеПосле подготовки linker передаёт управление entry point / startup-коду программы.
12

GOT и PLT

Как динамический линковщик подставляет адреса данных и функций

Global Offset Table (GOT)

GOT — таблица адресов, через которую код обращается к глобальным данным и некоторым динамически разрешаемым символам. Вместо того чтобы патчить много мест в коде, можно хранить адрес символа в одном месте — в GOT, а все обращения делать через неё.

🚀
Почему GOT ускоряет динамическую линковкуЕсли много инструкций используют один и тот же глобальный символ, динамическому линковщику не нужно исправлять каждую инструкцию. Достаточно один раз записать правильный адрес в слот GOT.

Procedure Linkage Table (PLT)

PLT используется для вызовов функций из динамических библиотек. Она содержит маленькие трамплины, которые вместе с GOT позволяют реализовать lazy binding: функция резолвится не при запуске программы, а при первом вызове.

1
Код вызывает функцию через PLTНапример, printf@plt, а не напрямую неизвестный адрес printf в libc.
2
PLT смотрит в соответствующий слот GOTИзначально этот слот может указывать не на настоящую функцию, а обратно в resolver-механику PLT/ld.so.
3
При первом вызове запускается resolverДинамический линковщик ищет настоящий адрес символа в нужной shared library.
4
GOT патчитсяВ слот GOT записывается настоящий адрес функции.
5
Последующие вызовы идут напрямуюPLT снова делает переход через GOT, но теперь GOT уже указывает на реальную функцию.
🧠
Зачем нужен трамплин в PLT?GOT — это данные: слот с адресом. PLT — это код: место, куда можно прыгнуть из инструкции call, и которое знает, как либо перейти по GOT, либо вызвать resolver с нужным индексом символа. Для lazy binding нужен не просто адрес, а логика первого вызова.
13

RELRO: RELocation Read-Only

Partial vs Full RELRO и связь с GOT/PLT

RELRO — защитный механизм, который делает некоторые области, связанные с релокациями, read-only после того, как динамический линковщик закончил нужные исправления. Это снижает риск атак, где злоумышленник перезаписывает GOT-слоты и перенаправляет вызовы функций.

РежимЧто происходитПлюсыМинусы
Partial RELROЧасть relocation-related секций становится read-only, но .got.plt обычно остаётся writable для lazy binding.Дешевле, совместимо с lazy binding.GOT-слоты PLT могут оставаться целью атак.
Full RELROВсе нужные динамические релокации, включая PLT-релокации, резолвятся при запуске, затем GOT становится read-only.Сильнее защита GOT.Старт программы может быть медленнее, потому что lazy binding отключается.
🔐
ФормулировкаFull RELRO требует заранее разрешить динамические символы, потому что после перевода GOT в read-only динамический линковщик уже не сможет лениво патчить GOT при первом вызове.
14

Что происходит при запуске исполняемого файла

execve, загрузка ELF, стек, ld.so, vvar/vdso/vsyscall

1
Процесс вызывает execveСтарое содержимое адресного пространства заменяется новым образом программы.
2
Ядро читает ELF header и program headersПроверяет формат, архитектуру, тип файла, entry point и список сегментов.
3
Ядро мапит LOAD-сегментыКод, read-only данные, data, bss-память получают виртуальные адреса и права доступа.
4
Создаётся пользовательский стекНа стек кладутся argc, argv, envp и auxiliary vector с информацией от ядра.
5
Мапятся специальные страницы ядраvvar, vdso, иногда vsyscall. Они ускоряют некоторые операции вроде получения времени без полноценного syscall.

Если есть INTERP, запускается dynamic linkerЯдро передаёт управление ld.so, а не сразу коду программы.
7
ld.so готовит динамическую линковкуИтерируется по DYNAMIC, мапит shared libraries, применяет релокации, подготавливает GOT/PLT, вызывает init hooks.
8
Запускается startup-код программыОбычно это не сразу main, а _start и runtime-библиотека, которая вызывает main(argc, argv, envp).
⚠️
main — не первая инструкцияВ C/C++ программа обычно начинает не с main. Сначала работает entry point _start, C runtime, dynamic linker, initialization hooks. Уже потом вызывается main.

Initialization и finalization hooks

Preinit, init, constructors, atexit, fini, destructors

До входа в main и после выхода из него может выполняться дополнительный код. Это нужно для runtime-библиотек, C++ global constructors/destructors, инициализации shared libraries, регистрации обработчиков завершения и т.д.

1
Preinit hooksОчень ранние хуки для основного executable. Используются редко.
2
Init hooks / constructorsИнициализация библиотек и C++ глобальных объектов до main.
3
main(argc, argv, envp)Основная пользовательская функция программы.
4
atexit handlersФункции, зарегистрированные через atexit(), вызываются при нормальном завершении.
5
Fini hooks / destructorsДеструкторы C++ глобальных объектов и финализация shared libraries.
Пример конструктора/деструктора в GCC/ClangC
__attribute__((constructor))
static void before_main(void) {
    // выполнится до main
}

__attribute__((destructor))
static void after_main(void) {
    // выполнится после нормального завершения
}

Мини-шпаргалка

Что обязательно помнить

  • Сборка C/C++ обычно состоит из preprocessing → compilation → assembly → linking.
  • Объектный файл уже содержит машинный код, но может иметь unresolved symbols и relocation entries.
  • ELF — формат для relocatable object files, executables, shared objects и core dumps.
  • Sections нужны линкеру: .text, .rodata, .data, .bss, symbol tables, relocation tables.
  • Segments нужны загрузчику: LOAD, INTERP, DYNAMIC, NOTE.
  • .bss описывает нулевую память, но не хранит огромный блок нулей в файле.
  • Релокации говорят, какие места в коде/данных нужно исправить адресами символов.
  • REL/RELA отличаются тем, где хранится addend: неявно в месте патча или явно в relocation entry.
  • PIC нужен для кода, который может быть загружен по разным адресам; PIE — position-independent executable.
  • ASLR рандомизирует адреса загрузки, поэтому часть адресов известна только при запуске.
  • Динамическая линковка позволяет разделять код shared libraries между процессами.
  • .interp указывает путь к динамическому линковщику, а .dynamic даёт ему metadata.
  • GOT хранит адреса динамически разрешаемых данных/символов в одном месте.
  • PLT содержит код-трамплины для вызова динамических функций и lazy binding.
  • Full RELRO делает GOT read-only после раннего разрешения релокаций, но отключает lazy binding.
  • main не первая инструкция: до него работают kernel loader, ld.so, _start, C runtime и init hooks.

Вопросы на понимание

Проверяют механику, а не запоминание названий

Q01
Почему «компиляция» в быту и compilation как стадия pipeline — не одно и то же?
В быту компиляцией называют весь путь до бинаря. Технически compilation — это перевод preprocessed source в assembly, а дальше есть assembly и linking.
Q02
Зачем нужен препроцессор?
Он раскрывает include-файлы, макросы и условную компиляцию, формируя translation unit для компилятора.
Q03
Почему объектный файл ещё не обязательно можно запустить?
В нём могут быть unresolved references и relocation entries. Он предназначен для линковки, а не прямого исполнения.
Q04
Чем секции ELF отличаются от сегментов?
Секции — логические области для линковки и анализа. Сегменты — области, которые загрузчик мапит в память при запуске.
Q05
Почему .bss не увеличивает файл так же сильно, как .data?
.data хранит начальные значения в файле, а .bss только описывает область, которую загрузчик заполнит нулями.
Q06
Что такое релокация простыми словами?
Это инструкция линкеру или динамическому линковщику: «вот в этом месте файла/памяти нужно записать адрес или смещение такого-то символа».
Q07
Почему часть релокаций нельзя вычислить при статической линковке?
Если используется PIE/ASLR или shared libraries, фактический load address известен только при запуске.
Q08
Зачем нужен PIC для shared libraries?
Одна shared library может быть загружена по разным адресам в разных процессах. Код должен работать независимо от базового адреса.
Q09
Что даёт PIE вместе с ASLR?
PIE позволяет рандомизировать базовый адрес самого исполняемого файла, а не только стек, heap и shared libraries.
Q10
Почему динамическая линковка экономит память?
Read-only код shared libraries может быть физически разделён между процессами, хотя маппится в их разные виртуальные адресные пространства.
Q11
Зачем нужна GOT?
Чтобы хранить динамически вычисленные адреса символов в одном месте и не патчить каждое использование символа в коде.
Q12
Зачем нужна PLT, если уже есть GOT?
GOT хранит адрес, а PLT содержит код-трамплин. Для lazy binding нужен код, который при первом вызове вызовет resolver с информацией о нужном символе, а затем прыгнет по обновлённой GOT.
Q13
Почему Full RELRO конфликтует с lazy binding?
Lazy binding требует патчить GOT при первом вызове. Full RELRO делает GOT read-only после начальной линковки, поэтому все PLT-релокации нужно разрешить заранее.
Q14
Почему main не является первой выполняемой функцией программы?
Сначала ядро загружает ELF, может запустить ld.so, затем работает _start и C runtime, вызываются init hooks, и только потом вызывается main.
Q15
Что ядро кладёт на пользовательский стек при запуске?
Аргументы командной строки, переменные окружения и auxiliary vector с системной информацией для runtime/linker.
Q16
Когда вызываются constructors и destructors?
Constructors/init hooks вызываются до main, destructors/fini hooks и atexit-handlers — при нормальном завершении после main.
Конспект покрывает путь от исходного C/C++ к исполняемому ELF, устройство ELF, статическую и динамическую линковку, GOT/PLT, RELRO и запуск программы через ядро, dynamic linker и runtime.
